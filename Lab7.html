Cloud Spanner System Tables & Statistics Lab

An intermediate-to-advanced production troubleshooting workshop designed for DBAs, Application Engineers, and Production Support Teams. Master INFORMATION_SCHEMA, SPANNER_SYS statistics tables, execution plans, and query profiles directly inside the Cloud Spanner Studio SQL Editor.

100% Console SQL Editor Executable
This entire lab is executed directly from the Google Cloud Spanner Console SQL Editor (Spanner Studio) using standard GoogleSQL DDL and DML statements. No Python scripts, Java drivers, or gcloud commands are required.

Target Audience & Prerequisites

  • Database Administrators (DBAs): Responsible for monitoring query performance, lock contention, column operation statistics, and capacity planning in multi-region Spanner clusters.
  • Application & Software Engineers: Building high-throughput core banking and payments APIs that require sub-millisecond point reads and deadlock-free transactions.
  • Production Support & SRE Engineers: Diagnosing live production incidents, high CPU alerts, commit latency spikes, and slow query alerts.

System & Statistics Tables Overview

Google Cloud Spanner exposes real-time runtime metrics through two primary system schemas:

INFORMATION_SCHEMA
Contains metadata about database structure: tables, columns, indexes, foreign keys, table interleaving, check constraints, and database options. Read-only and instantaneous.
SPANNER_SYS Schema
Contains aggregated operational metrics over 1-minute (`_TOP_MINUTE`), 10-minute (`_TOP_10MINUTE`), and 1-hour (`_TOP_HOUR`) time windows: query statistics, lock waits, transaction commit latencies, and column read/write operation counts.
Official Retention Windows & Collection Delays
1-Minute Tables (`_TOP_MINUTE`): Retained for 6 hours.
10-Minute Tables (`_TOP_10MINUTE`): Retained for 4 days.
1-Hour Tables (`_TOP_HOUR`): Retained for 30 days.
Collection Delay: 1-minute statistics tables typically lag live execution by 1 to 3 minutes.
Local Emulator Note: The Spanner local emulator supports `INFORMATION_SCHEMA` queries fully, but background `SPANNER_SYS` metrics tables are populated primarily in GCP Cloud Spanner environments.
  • Review Lab Architecture & System Table Schemas Understand the operational boundary between INFORMATION_SCHEMA metadata and SPANNER_SYS metrics tables.

Step 2: Deploy the Banking & Payments DDL Schema

In this step, you will create a production-grade Banking database schema containing Customers, Accounts, Transactions, Merchants, and AuditEvents directly from the Spanner Studio SQL Editor.

Handling Existing Tables ("Duplicate name in schema")
If your database already contains tables from a previous execution, running `CREATE TABLE` will raise Statement failed: Duplicate name in schema: Customers.

Solutions:
  1. Use CREATE TABLE IF NOT EXISTS (included in the script below) to skip existing tables safely.
  2. If you want to re-create the schema from scratch, drop existing indexes and tables in indexes-first, child-first order (because child tables and indexes depend on parent objects): 
    DROP INDEX TransactionsByMerchantDateCovered;
DROP INDEX TxnByMerchantStatus;
DROP INDEX AccountsByStatusCovered;
DROP INDEX AccountsByStatus;
DROP TABLE Transactions;
DROP TABLE Accounts;
DROP TABLE Customers;
DROP TABLE Merchants;
DROP TABLE AuditEvents;
GoogleSQL DDL (Execute in Spanner Studio)
Copied! 
-- 1. Customers Table (Parent)
CREATE TABLE IF NOT EXISTS Customers (
  CustomerId   INT64 NOT NULL,
  FullName     STRING(255) NOT NULL,
  Email        STRING(255) NOT NULL,
  RiskScore    FLOAT64,
  KycStatus    STRING(50) NOT NULL,
  CreatedAt    TIMESTAMP NOT NULL
) PRIMARY KEY (CustomerId);

-- 2. Accounts Table (Child interleaved in Customers)
CREATE TABLE IF NOT EXISTS Accounts (
  CustomerId   INT64 NOT NULL,
  AccountId    INT64 NOT NULL,
  AccountType  STRING(50) NOT NULL,
  Currency     STRING(10) NOT NULL,
  Balance      NUMERIC NOT NULL,
  Status       STRING(30) NOT NULL,
  UpdatedAt    TIMESTAMP NOT NULL
) PRIMARY KEY (CustomerId, AccountId),
  INTERLEAVE IN PARENT Customers ON DELETE CASCADE;

-- 3. Transactions Table (Grandchild interleaved in Accounts)
CREATE TABLE IF NOT EXISTS Transactions (
  CustomerId      INT64 NOT NULL,
  AccountId       INT64 NOT NULL,
  TransactionId   INT64 NOT NULL,
  MerchantId      INT64,
  Amount          NUMERIC NOT NULL,
  TxnType         STRING(50) NOT NULL,
  TxnTimestamp    TIMESTAMP NOT NULL,
  Status          STRING(30) NOT NULL
) PRIMARY KEY (CustomerId, AccountId, TransactionId),
  INTERLEAVE IN PARENT Accounts ON DELETE CASCADE;

-- 4. Merchants Standalone Table
CREATE TABLE IF NOT EXISTS Merchants (
  MerchantId      INT64 NOT NULL,
  MerchantName    STRING(255) NOT NULL,
  Category        STRING(100) NOT NULL,
  CountryCode     STRING(10) NOT NULL,
  IsActive        BOOL NOT NULL
) PRIMARY KEY (MerchantId);

-- 5. AuditEvents Table (Hotspot intentional bad primary key design)
CREATE TABLE IF NOT EXISTS AuditEvents (
  EventTimestamp  TIMESTAMP NOT NULL,
  EventId         INT64 NOT NULL,
  UserId          STRING(100),
  Action          STRING(100),
  Payload         STRING(MAX)
) PRIMARY KEY (EventTimestamp, EventId);

-- 6. Secondary Index on Accounts by Status
CREATE INDEX IF NOT EXISTS AccountsByStatus ON Accounts(Status);

Step 4: INFORMATION_SCHEMA — Inspecting Table Hierarchy & Interleaving

Investigate parent-child interleave relationships and primary key definitions using system metadata views.

Objective
Verify table interleaving structure and confirm `ON DELETE CASCADE` properties without reading full schema DDL files.
Business Scenario
A production engineer needs to audit whether `Transactions` table deletes will clean up child rows automatically during account closure.
GoogleSQL Query (Execute in Spanner Studio)
Copied! 
SELECT 
  table_name, 
  parent_table_name, 
  on_delete_action
FROM INFORMATION_SCHEMA.TABLES
WHERE table_schema = ''
ORDER BY table_name;
Expected Observations & DBA Interpretation

Observations: Accounts shows parent_table_name = 'Customers' with on_delete_action = 'CASCADE'. Transactions shows parent_table_name = 'Accounts' with on_delete_action = 'CASCADE'.

DBA Takeaway: Interleaved tables physically co-locate child rows alongside parent rows inside Spanner splits. A delete on a parent row with CASCADE will automatically delete all child rows in a single atomic transaction without cross-split RPC overhead.

  • Run Table Interleave Query Verify that parent_table_name and on_delete_action reflect the expected schema hierarchy.

Step 5: INFORMATION_SCHEMA — Auditing Index Definitions & Covered Columns

Discover secondary index states, indexed column order, and stored columns using `INFORMATION_SCHEMA.INDEXES` and `INDEX_COLUMNS`.

GoogleSQL Query (Execute in Spanner Studio)
Copied! 
SELECT 
  i.table_name,
  i.index_name,
  i.index_type,
  i.is_unique,
  i.index_state,
  ic.column_name,
  ic.ordinal_position,
  ic.column_ordering
FROM INFORMATION_SCHEMA.INDEXES i
JOIN INFORMATION_SCHEMA.INDEX_COLUMNS ic
  ON i.table_name = ic.table_name AND i.index_name = ic.index_name
WHERE i.table_schema = '' AND i.index_name != 'PRIMARY_KEY'
ORDER BY i.table_name, i.index_name, ic.ordinal_position;
Expected Observations & Interpretation

Observations: Shows AccountsByStatus with index_state = 'READ_ONLY' (ready for production queries). If an index is currently being built in background, its state displays as WRITE_ONLY or BUILDING.

DBA Takeaway: Never route live production traffic to a secondary index until index_state reaches READ_ONLY in INFORMATION_SCHEMA.INDEXES.

  • Execute Index Metadata Audit Query Confirm that secondary index status shows READ_ONLY.

Step 6: SPANNER_SYS — Identifying Top CPU-Consuming Queries

Locate high-cost SQL statements responsible for elevated CPU utilization using `SPANNER_SYS.QUERY_STATS_TOP_MINUTE`.

Objective
Find queries consuming the highest cumulative CPU seconds over recent 1-minute aggregation windows.
Business Scenario
Cloud Monitoring triggers an alert: Spanner High CPU Usage (>85%). The DBA must instantly identify the exact query fingerprint causing node saturation.

Phase 1 — Execute Heavy CPU Workload Queries

Run the following intentionally problematic queries in Spanner Studio 2–3 times. These queries perform un-indexed substring scans, cross joins, and string hashing that generate elevated CPU consumption on your Spanner instance:

GoogleSQL Heavy Workload Queries (Execute in Spanner Studio)
Copied! 
-- Heavy Query 1: Un-indexed Cross Join & Substring Regex Match
SELECT 
  c.CustomerId, 
  c.FullName, 
  c.Email, 
  a.Balance, 
  FARM_FINGERPRINT(CONCAT(c.FullName, c.Email, CAST(a.Balance AS STRING))) AS HeavyHash
FROM Customers c
CROSS JOIN Accounts a
WHERE LOWER(c.Email) LIKE '%apex%' OR LOWER(c.FullName) LIKE '%vance%'
LIMIT 500;

-- Heavy Query 2: Un-indexed Aggregation with String Hashing & Sorting
SELECT 
  UserId, 
  Action, 
  COUNT(*) AS ActionCount,
  MAX(FARM_FINGERPRINT(Payload)) AS MaxPayloadHash
FROM AuditEvents
WHERE LOWER(Payload) LIKE '%transfer%' OR LOWER(Action) LIKE '%login%'
GROUP BY UserId, Action
ORDER BY ActionCount DESC;

-- Heavy Query 3: Multi-table Join Filtering on Un-indexed TxnTimestamp & Amount Ranges
SELECT 
  t.CustomerId, 
  t.AccountId, 
  t.TransactionId, 
  t.Amount, 
  m.MerchantName, 
  c.FullName
FROM Transactions t
JOIN Merchants m ON t.MerchantId = m.MerchantId
JOIN Customers c ON t.CustomerId = c.CustomerId
WHERE t.Amount > 10.00 AND LOWER(m.Category) LIKE '%retail%';

Phase 2 — Inspect Top CPU Consuming Queries via SPANNER_SYS

After running the workload queries above, execute this query to query SPANNER_SYS.QUERY_STATS_TOP_MINUTE and discover which statement generated the highest CPU footprint:

Filtering System Queries, Ping Checks (`SELECT 1`), & Collection Delays
1. Filtering Internal System Queries & Health Pings (`SELECT 1`): Without filtering, QUERY_STATS_TOP_MINUTE includes Spanner Studio console calls, metadata queries, and connection pool keep-alive pings (e.g. SELECT 1). Adding filters for spanner_sys, information_schema, and select 1 isolates true business application workload queries.

2. Aggregation Window Delay (1–3 Minutes): Spanner collects statistics in 1-minute clock-aligned intervals. If you just executed a query, wait 1 to 3 minutes for the current 1-minute interval boundary to close and populate in SPANNER_SYS.QUERY_STATS_TOP_MINUTE.
GoogleSQL Query (Execute in Spanner Studio)
Copied! 
SELECT 
  text_fingerprint,
  text,
  execution_count,
  avg_cpu_seconds,
  (execution_count * avg_cpu_seconds) AS total_cpu_seconds,
  avg_latency_seconds,
  avg_rows_scanned,
  avg_rows,
  avg_bytes
FROM SPANNER_SYS.QUERY_STATS_TOP_MINUTE
WHERE LOWER(text) NOT LIKE '%spanner_sys%'
  AND LOWER(text) NOT LIKE '%information_schema%'
  AND LOWER(text) NOT LIKE '%select 1%'
  AND LOWER(text) NOT LIKE '%select current_timestamp%'
ORDER BY total_cpu_seconds DESC
LIMIT 10;
How to Interpret Column Metrics
  • total_cpu_seconds: The ultimate metric for identifying queries starving CPU resources. A query executed 1,000 times consuming 5ms per call (5.0s total CPU) demands more attention than a query executed once taking 2.0s.
  • text_fingerprint: Unique hash identifier for parametrized SQL templates. Use this fingerprint in Query Insights for trend analysis across hours/days.
  • Execute Top CPU Query Stats Inspection Identify the query text with highest total_cpu_seconds.

Step 8: SPANNER_SYS — High Rows Scanned vs. Rows Returned (Scan Efficiency Ratio)

Detect unindexed full table scans and inefficient filtering using `avg_rows_scanned` and `avg_rows`.

Objective
Calculate the ratio of `avg_rows_scanned / avg_rows` to find queries performing heavy scanning for minimal output.
Business Scenario
A fraud monitoring background query is scanning 500,000 transaction records to return only 3 matching suspicious rows.
GoogleSQL Query (Execute in Spanner Studio)
Copied! 
SELECT 
  text_fingerprint,
  text,
  avg_rows_scanned,
  avg_rows,
  SAFE_DIVIDE(avg_rows_scanned, avg_rows) AS scan_efficiency_ratio,
  avg_bytes,
  avg_cpu_seconds
FROM SPANNER_SYS.QUERY_STATS_TOP_MINUTE
WHERE avg_rows > 0
  AND LOWER(text) NOT LIKE '%spanner_sys%'
  AND LOWER(text) NOT LIKE '%information_schema%'
  AND LOWER(text) NOT LIKE '%select 1%'
  AND LOWER(text) NOT LIKE '%select current_timestamp%'
ORDER BY scan_efficiency_ratio DESC
LIMIT 10;
How to Interpret Scan Efficiency Ratios

Ideal Ratio: ~1.0 (Direct index seek returning exact target rows).
Warning Ratio (>100.0): Indicates Spanner is scanning 100+ storage rows to find 1 matching row due to unindexed columns or leading non-selective index keys.

  • Run Scan Efficiency Ratio Query Find queries where scanned rows dramatically outnumber returned rows.

Step 10: SPANNER_SYS — Column Operation Statistics & Hotspot Detection

Inspect read, query, and write traffic distribution across table columns using `SPANNER_SYS.COLUMN_OPERATIONS_STATS_MINUTE`.

GoogleSQL Query (Execute in Spanner Studio)
Copied! 
SELECT 
  table_name,
  column_name,
  read_count,
  query_count,
  write_count,
  (read_count + query_count + write_count) AS total_ops
FROM SPANNER_SYS.COLUMN_OPERATIONS_STATS_MINUTE
ORDER BY total_ops DESC
LIMIT 15;
Production Interpretation: Column-Level Traffic & Hotspotting

Columns with massive write_count (e.g. AuditEvents.EventTimestamp) using sequential values cause split hotspotting. Fix by using UUIDs or Bit-Reversed sequences for primary key columns.

  • Run Column Operation Stats Query Verify which columns incur the highest read, query, and write operation counts.

Step 11: SPANNER_SYS — Lock Statistics & Lock Wait Duration

Investigate database lock contention and blocked transactions using `SPANNER_SYS.LOCK_STATS_TOP_MINUTE`.

Objective
Measure `lock_wait_seconds` and lock wait frequencies to detect concurrent transaction blocking.
Business Scenario
During peak payment processing hours, transfer transactions report 3-second latencies caused by lock contention on VIP merchant account rows.
GoogleSQL Query (Execute in Spanner Studio)
Copied! 
SELECT 
  interval_end,
  row_range_start_key,
  lock_wait_seconds
FROM SPANNER_SYS.LOCK_STATS_TOP_MINUTE
WHERE lock_wait_seconds > 0
ORDER BY lock_wait_seconds DESC;
Understanding Spanner Lock Modes & Pessimistic Locking

Lock statistics are populated for read-write transactions acquiring pessimistic locks.
DBA Best Practice: If read-write transactions execute long SQL selects before mutating, convert those read operations to a separate ReadOnlyTransaction or snapshot read to avoid holding shared locks!

  • Run Lock Statistics Inspection Verify lock_wait_seconds and row_range_start_key across top lock contention intervals.

Step 12: SPANNER_SYS — Diagnosing Row Key & Column Contention

Inspect `SAMPLE_LOCK_REQUESTS` array of structs in `LOCK_STATS_TOP_MINUTE` to find exact columns and lock modes involved in conflicts.

GoogleSQL Query (Execute in Spanner Studio)
Copied! 
SELECT 
  interval_end,
  lock_wait_seconds,
  req.column,
  req.lock_mode,
  req.transaction_tag
FROM SPANNER_SYS.LOCK_STATS_TOP_MINUTE,
UNNEST(sample_lock_requests) AS req
ORDER BY lock_wait_seconds DESC
LIMIT 10;
Cell-Level Locking Deep Dive

Cloud Spanner locks individual cells (column values) rather than entire rows when possible. However, primary key columns represent row identity — mutating a primary key column or acquiring exclusive locks on non-interleaved parent-child tables will lock the entire key space for that row.

  • Inspect Sample Lock Requests via UNNEST Examine sampled columns, lock modes (ReaderShared, WriterShared), and transaction tags.

Step 13: SPANNER_SYS — Transaction Total Latency & Commit Attempts

Analyze transaction performance, commit attempts, and total latency using `SPANNER_SYS.TXN_STATS_TOP_MINUTE`.

GoogleSQL Query (Execute in Spanner Studio)
Copied! 
SELECT 
  fprint,
  transaction_tag,
  commit_attempt_count,
  avg_total_latency_seconds,
  avg_bytes
FROM SPANNER_SYS.TXN_STATS_TOP_MINUTE
ORDER BY avg_total_latency_seconds DESC;
Single-Split vs. Multi-Split (2PC) Commit Latencies

Single Split Commit: Spanner completes commit using Paxos consensus within a single split (~5-15ms latency).
Multi-Split 2PC Commit: Spanner must run Two-Phase Commit across multiple split leaders when mutating multiple non-interleaved tables (50-200ms latency). Interleaving child tables keeps related rows in a single split!

  • Run Transaction Stats Query Compare avg_total_latency_seconds against commit_attempt_count and transaction_tag.

Step 14: SPANNER_SYS — Average Bytes & Table Operation Impact

Inspect `OPERATIONS_BY_TABLE` array in `TXN_STATS_TOP_MINUTE` to find which tables are impacted by top transactions.

GoogleSQL Query (Execute in Spanner Studio)
Copied! 
SELECT 
  interval_end,
  fprint,
  transaction_tag,
  avg_bytes,
  ops.table_name,
  ops.insert_or_update_count,
  ops.insert_or_update_bytes
FROM SPANNER_SYS.TXN_STATS_TOP_MINUTE,
UNNEST(operations_by_table) AS ops
ORDER BY avg_bytes DESC
LIMIT 10;
Transaction Tagging Best Practices

Always tag transactions in application code (e.g. TransactionOptions.withTag("checkout-payment")). When transaction tags are present, SPANNER_SYS.TXN_STATS groups metrics by tag name, allowing instant identification of offending microservices.

  • Unnest Operations by Table in TXN_STATS Identify the specific tables incurring writes and heavy byte payloads in top transactions.

Step 17: Enterprise Incident Case Studies

Review 5 real-world production incident scenarios resolved using system tables and index optimization.

Case 1: Midnight Batch Run Spiking CPU
Root Cause: Un-indexed date filter on AuditEvents table.
Solution: Added null-filtered secondary index on event timestamp.
Case 2: Payment Gateway 2PC Commitment Latency
Root Cause: Updating non-interleaved accounts table across 4 splits.
Solution: Re-architected schema to interleave Accounts inside Customers.
Case 3: Hotspotting on Sequential Primary Keys
Root Cause: Timestamp-leading PK on AuditEvents causing single split overload.
Solution: Prefixed PK with FARM_FINGERPRINT(UserId) hash.
Case 4: High Transaction Abort Spike
Root Cause: Long read-select statement inside read-write transaction holding shared locks.
Solution: Moved read queries out of RW transaction into snapshot read.
Case 5: Slow Customer Search API
Root Cause: Base table lookup fetching KycStatus for every search result.
Solution: Added STORING (KycStatus) to Customer search secondary index.

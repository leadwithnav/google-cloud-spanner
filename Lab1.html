Provisioning Cloud Spanner & Running the Emulator

Welcome! In this hands-on lab, you will learn how to configure the Google Cloud Spanner local emulator, provision Spanner instances & databases, define schemas using Data Definition Language (DDL), and manipulate data using Data Manipulation Language (DML).

What is Google Cloud Spanner?

Google Cloud Spanner is an enterprise-grade, globally distributed, horizontally scalable, and strongly consistent relational database service. It combines the benefits of a relational database structure (schemas, ACID transactions, SQL queries) with the scalability of a non-relational (NoSQL) database.

Key Spanner Concepts
Unlike typical relational databases, Spanner scales horizontally to thousands of nodes. It manages replication automatically and handles splits (shards of tables) transparently based on key ranges.

Prerequisites

Before proceeding, make sure you have the following installed on your machine:

  • Google Cloud SDK (gcloud CLI): Used for executing database administrative commands.
  • Docker: Required if you choose to run the Cloud Spanner Emulator via container (Recommended).
Emulator Sandbox Behavior
The local Cloud Spanner emulator is meant for prototyping and local testing. It runs entirely in-memory, meaning that restarting the emulator will wipe out any created instances, schemas, and records.

Step 1: Starting the Cloud Spanner Emulator

There are two primary methods to run the local Cloud Spanner emulator: using **Docker** or using the **gcloud CLI components**. Select the tab that corresponds to your preferred environment.

Using Docker runs the database emulator inside a lightweight container, keeping your system cleanly separated. Run this command in your terminal:

Terminal (Bash / PowerShell)
Copied! 
docker run -d -p 9010:9010 -p 9020:9020 \
  --name spanner-emulator \
  gcr.io/cloud-spanner-emulator/emulator

This command pulls the emulator image from the Google Container Registry, maps ports 9010 (gRPC API) and 9020 (REST API) to your host computer, and runs it as a background service named spanner-emulator.

If you don't have Docker installed, you can launch the emulator directly from your local Google Cloud SDK directory. Run these commands:

Terminal (Bash / PowerShell)
Copied! 
# Install the Spanner Emulator component
gcloud components install cloud-spanner-emulator

# Start the emulator
gcloud emulators spanner start
Blocking Command Warning
Running gcloud emulators spanner start blocks your current terminal window. Keep it running and open a new terminal tab to run the remaining configuration commands.
  • Start the Cloud Spanner Emulator container or process Execute the command to launch the emulator and verify it runs on port 9010.
  • Verify emulator ports are listening Ensure localhost:9010 is reachable from your local shell session.

Step 2: Configuring gcloud CLI to Use the Emulator

By default, the gcloud command-line tool sends API commands to the global, production Google Cloud endpoints. To redirect your administrative commands to the local emulator, you must configure a dedicated gcloud environment profile.

Create an Emulator Profile

Rather than modifying your default cloud profile, we will create a brand new configuration called emulator. Run the following commands:

Terminal (Bash / PowerShell)
Copied! 
# Create a new local gcloud configuration profile
gcloud config configurations create emulator

# Set the endpoint override to route Spanner commands to the local emulator
gcloud config set api_endpoint_overrides/spanner http://localhost:9020/

# Set a dummy project ID (required by gcloud commands but ignored by the emulator)
gcloud config set project emulator-project

# Disable credentials since the local emulator doesn't require authentication
gcloud config set auth/disable_credentials true
Switching Back to Production
You can switch back to your production configurations anytime with:
gcloud config configurations activate [YOUR_DEFAULT_CONFIG]

Verify Configuration Settings

Confirm that your active configuration parameters match the local emulator parameters. Run this command:

Terminal (Bash / PowerShell)
Copied! 
gcloud config list
  • Create the emulator configuration profile Initialize the 'emulator' profile under gcloud CLI configurations.
  • Configure the endpoint overrides, project and credentials Set project ID to 'emulator-project' and disable auth credentials.

Step 3: Provisioning Spanner Instance & Database

In Google Cloud Spanner, resources are organized hierarchically: Project → Instance → Database → Tables. An instance defines the compute allocation (nodes or processing units) and configuration. The database stores the schema and table content.

1. Provision a Spanner Instance

Create a local instance called test-instance. We'll set the configuration config value to emulator-config (which tells the CLI it is an emulator instance) and provision it with 1 node:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner instances create test-instance \
  --config=emulator-config \
  --description="Local Development Instance" \
  --nodes=1

2. Create a Database

Now, create a database named test-database associated with your newly initialized instance:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases create test-database \
  --instance=test-instance

3. Verify Created Resources

List your instances and databases to verify that they were successfully provisioned in the emulator:

Terminal (Bash / PowerShell)
Copied! 
# List all instances
gcloud spanner instances list

# List databases in test-instance
gcloud spanner databases list --instance=test-instance
  • Create the test-instance Provision test-instance with config 'emulator-config'.
  • Create the test-database Link the new database 'test-database' to 'test-instance'.

Step 4: Defining the Schema (DDL)

In Google Cloud Spanner, you use standard Data Definition Language (DDL) commands to create and edit tables. Each table in Spanner must have a primary key specified to identify rows and guide spatial data distribution.

We will create two independent simple tables: TradingAccounts (to store account profiles) and Trades (to store trade transactions).

Execute the DDL Command

Write the schema changes to your database instance using the gcloud spanner databases ddl update tool:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database \
  --instance=test-instance \
  --ddl='CREATE TABLE TradingAccounts (
    AccountId INT64 NOT NULL,
    AccountName STRING(1024),
    FirmName STRING(1024),
    AccountInfo BYTES(MAX)
  ) PRIMARY KEY (AccountId);
  CREATE TABLE Trades (
    AccountId INT64 NOT NULL,
    TradeId INT64 NOT NULL,
    ProductSymbol STRING(MAX),
    NotionalValue INT64
  ) PRIMARY KEY (AccountId, TradeId);'

Check Schema Integrity

Query the database's metadata schema details to verify that both tables are successfully registered:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl describe test-database \
  --instance=test-instance
  • Understand Primary Key layouts Verify that each table specifies a primary key to guide data sharding.
  • Execute DDL to define the tables Apply the CREATE TABLE statements inside the test-database schema.

Step 5: Executing DML and Queries

In Google Cloud Spanner, you modify table records using standard relational Data Manipulation Language (DML) statements such as INSERT, UPDATE, and DELETE. To run SQL statements via the command line, use `execute-sql`.

1. Insert Data (DML INSERT)

Run these commands to insert sample records into the TradingAccounts and Trades tables:

Terminal (Bash / PowerShell)
Copied! 
# Insert a record into the TradingAccounts table
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="INSERT INTO TradingAccounts (AccountId, AccountName, FirmName) VALUES (1, 'CME_MAIN_ACC', 'CME Group Client Services')"

# Insert a record into the Trades table
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="INSERT INTO Trades (AccountId, TradeId, ProductSymbol, NotionalValue) VALUES (1, 101, 'ESU6 (E-mini S&P 500 Futures)', 5000000)"

2. Modify Data (DML UPDATE)

Update a specific row in the database table using DML:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="UPDATE Trades SET NotionalValue = 6500000 WHERE AccountId = 1 AND TradeId = 101"

3. Run a Query (SELECT)

Run a cross-table SQL query using a JOIN operation to fetch records from both tables:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="SELECT a.AccountName, a.FirmName, t.ProductSymbol, t.NotionalValue FROM TradingAccounts AS a JOIN Trades AS t ON a.AccountId = t.AccountId"
  • Run DML commands to populate data Execute the INSERT statements for the TradingAccount and Trade rows.
  • Verify changes with SELECT statements Verify that the returned query contains updated values.

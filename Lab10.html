Step 1: Cloud Spanner Geo-Partitioning Concepts

Understand row-level data partitioning, placement keys, and multi-region storage routing in Cloud Spanner using GoogleSQL & gcloud.

🌐 What is Row-Level Geo-Partitioning?
Cloud Spanner Geo-Partitioning allows database administrators to pin individual database rows to specific geographic regions (e.g. US-East, US-West, Europe-West) using a designated PLACEMENT KEY column. This guarantees low-latency local reads/writes while ensuring compliance with regional data residency requirements (e.g. GDPR, CCPA).
⚡ Database Placement Architecture (1 Partition Setup)
placement_us_east
Instance Partition: us-east1-partition
Stores US-East customer rows with local single-digit millisecond latency.
DEFAULT Placement
Base Multi-Region Split
Stores rows without custom placement boundaries.

Step 2: Creating Spanner Instance & Instance Partition

Use gcloud CLI to provision a multi-region instance and create 1 regional instance partition for database placements.

💡 What is an Instance Partition?
An Instance Partition is a dedicated pool of compute and storage resources (processing units or nodes) residing in a specific region inside your Spanner instance.
Linux / macOS Commands
# 1. Create Multi-Region Spanner Instance (nam6 configuration)
gcloud spanner instances create geo-spanner-instance \
    --config=nam6 \
    --description="Geo Spanner Instance" \
    --edition=ENTERPRISE_PLUS \
    --nodes=1

# 2. Create Single Regional Instance Partition (us-east1-partition)
gcloud spanner instance-partitions create us-east1-partition \
    --instance=geo-spanner-instance \
    --config=regional-us-east1 \
    --description="US East Partition" \
    --nodes=1
Windows PowerShell Commands
# 1. Create Multi-Region Spanner Instance (nam6 configuration)
gcloud spanner instances create geo-spanner-instance `
    --config=nam6 `
    --description="Geo Spanner Instance" `
    --edition=ENTERPRISE_PLUS `
    --nodes=1

# 2. Create Single Regional Instance Partition (us-east1-partition)
gcloud spanner instance-partitions create us-east1-partition `
    --instance=geo-spanner-instance `
    --config=regional-us-east1 `
    --description="US East Partition" `
    --nodes=1

Step 4: DDL Syntax – CREATE PLACEMENT

Define database placement object mapping logical placement to physical instance partition.

GoogleSQL Dialect Placement DDL
-- Define Regional Database Placement (placement_us_east -> us-east1-partition)
CREATE PLACEMENT IF NOT EXISTS placement_us_east OPTIONS (instance_partition = 'us-east1-partition');
gcloud DDL Command
gcloud spanner databases ddl update geo-bank-db \
  --instance=geo-spanner-instance \
  --ddl="CREATE PLACEMENT IF NOT EXISTS placement_us_east OPTIONS (instance_partition = 'us-east1-partition')"
gcloud DDL Command (PowerShell)
gcloud spanner databases ddl update geo-bank-db `
  --instance=geo-spanner-instance `
  --ddl="CREATE PLACEMENT IF NOT EXISTS placement_us_east OPTIONS (instance_partition = 'us-east1-partition')"
  • Inspect Placement DDL Verify OPTIONS (instance_partition = '...') syntax for Spanner GoogleSQL dialect.

Step 5: DDL Syntax – PLACEMENT KEY & Table Schema

Decorate table column with PLACEMENT KEY to enable row-level regional data routing.

GoogleSQL Dialect Table Schema DDL
CREATE TABLE IF NOT EXISTS user_geo_accounts (
    user_id STRING(36) NOT NULL,
    region_location STRING(MAX) NOT NULL PLACEMENT KEY,
    country_code STRING(2) NOT NULL,
    full_name STRING(255) NOT NULL,
    email STRING(255) NOT NULL,
    account_type STRING(50) NOT NULL,
    balance NUMERIC NOT NULL,
    created_at TIMESTAMP NOT NULL OPTIONS (allow_commit_timestamp = true)
) PRIMARY KEY (user_id);
gcloud DDL Command
gcloud spanner databases ddl update geo-bank-db \
  --instance=geo-spanner-instance \
  --ddl="CREATE TABLE IF NOT EXISTS user_geo_accounts (user_id STRING(36) NOT NULL, region_location STRING(MAX) NOT NULL PLACEMENT KEY, country_code STRING(2) NOT NULL, full_name STRING(255) NOT NULL, email STRING(255) NOT NULL, account_type STRING(50) NOT NULL, balance NUMERIC NOT NULL, created_at TIMESTAMP NOT NULL OPTIONS (allow_commit_timestamp = true)) PRIMARY KEY (user_id);"
gcloud DDL Command (PowerShell)
gcloud spanner databases ddl update geo-bank-db `
  --instance=geo-spanner-instance `
  --ddl="CREATE TABLE IF NOT EXISTS user_geo_accounts (user_id STRING(36) NOT NULL, region_location STRING(MAX) NOT NULL PLACEMENT KEY, country_code STRING(2) NOT NULL, full_name STRING(255) NOT NULL, email STRING(255) NOT NULL, account_type STRING(50) NOT NULL, balance NUMERIC NOT NULL, created_at TIMESTAMP NOT NULL OPTIONS (allow_commit_timestamp = true)) PRIMARY KEY (user_id);"
  • Inspect Schema DDL Confirm user_id PRIMARY KEY and region_location STRING(MAX) NOT NULL PLACEMENT KEY in GoogleSQL.

Step 11: Scaling & Deleting Instance Partitions

Dynamically scale compute nodes or remove regional instance partitions using gcloud CLI.

📈 1. Scaling Compute Capacity Online
You can scale compute nodes of an instance partition up or down without downtime using gcloud spanner instance-partitions update.
Linux / macOS Commands
# 1. Scale Compute Nodes of an Instance Partition (e.g. from 1 node to 2 nodes)
gcloud spanner instance-partitions update us-east1-partition \
    --instance=geo-spanner-instance \
    --nodes=2

# 2. Precondition to Delete Partition: Drop Placement First
gcloud spanner databases ddl update geo-bank-db \
    --instance=geo-spanner-instance \
    --ddl="DROP PLACEMENT placement_us_east"

# 3. Delete Instance Partition
gcloud spanner instance-partitions delete us-east1-partition \
    --instance=geo-spanner-instance
Windows PowerShell Commands
# 1. Scale Compute Nodes of an Instance Partition (PowerShell)
gcloud spanner instance-partitions update us-east1-partition `
    --instance=geo-spanner-instance `
    --nodes=2

# 2. Precondition to Delete Partition: Drop Placement First (PowerShell)
gcloud spanner databases ddl update geo-bank-db `
    --instance=geo-spanner-instance `
    --ddl="DROP PLACEMENT placement_us_east"

# 3. Delete Instance Partition (PowerShell)
gcloud spanner instance-partitions delete us-east1-partition `
    --instance=geo-spanner-instance

Step 12: GoogleSQL vs PostgreSQL Dialect DDL

Compare DDL syntax differences for placements between GoogleSQL and PostgreSQL dialects.

📊 Dialect Syntax Comparison
  • GoogleSQL Dialect: CREATE PLACEMENT placement_name OPTIONS (instance_partition="partition_id");
  • PostgreSQL Dialect: CREATE PLACEMENT placement_name WITH (instance_partition='partition_id');
  • Covering Index: GoogleSQL uses STORING (...), while PostgreSQL uses INCLUDE (...).

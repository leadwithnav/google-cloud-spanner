Deploying Multi-Region Instances & Point-in-Time Recovery

Welcome! In this advanced lab, you will learn how to design disaster recovery models with Google Cloud Spanner. You will configure multi-region replica architectures and leverage Point-in-Time Recovery (PITR) to restore tables from logical data corruption.

High Availability in Spanner

Google Cloud Spanner guarantees high availability by replicating data across multiple zones (in a single region) or multiple regions. Regional outages are handled automatically by Spanner using dynamic synchronous replication and Paxos consensus routing. Even during a complete zone or regional failure, Spanner continues reading and writing transactions without manual failover commands, guaranteeing an industry-leading 99.999% SLA.

Outage Failure vs. Logical Failure

It is critical to distinguish between infrastructure failures and operational failures:

Infrastructure Failures
Hardware crashes, power cuts, or fiber lines cut. Spanner routes transactions around failed hardware instantly via Paxos. Mitigated automatically.
Logical Data Corruption
Accidental deletions, faulty applications writing garbage data, or malicious hacks. Because Spanner replicates immediately, the deleted data is wiped everywhere. Mitigated using PITR.

Point-in-Time Recovery (PITR)

PITR allows you to query historical data in Spanner at a microsecond scale in the past. If a developer accidentally deletes a table's records, Spanner can restore or query the database as it existed *before* the deletion, up to a maximum retention period of 7 days.

Step 1: Deploying a Multi-Region Instance

In production, you can choose between a single region (99.99% SLA) and a multi-region configuration (99.999% SLA). Multi-region setups involve multiple read-write replicas, read-only replicas, and witness nodes across geographically separated regional locations.

1. Explore Available Multi-Region Configurations

You can list the regional profiles supported by Google Cloud. Run this command in your cloud shell terminal:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner instance-configs list \
  --filter="name:nam"

This filters configuration templates like nam3 (multi-region North America configuration containing regions Iowa, South Carolina, and Oregon) and eur5 (multi-region Europe configuration).

2. Deploy the Multi-Region Instance

Create a production-grade instance named multi-region-spanner using the nam3 configuration template:

Terminal (Bash / PowerShell)
Copied! 
# Note: Running this command in production starts cloud billing costs
gcloud spanner instances create multi-region-spanner \
  --config=nam3 \
  --description="HA Multi-Region Production Instance" \
  --nodes=1
Emulator Support Notice
Because the local Spanner emulator runs in-memory on port 9010, it replicates all database commands locally. For the remainder of this training lab, we will use our active test-instance on the local emulator.

Step 2: Configuring Version Retention (PITR)

To recover data from a point in the past, Spanner must store database historical states. This is managed by setting the version_retention_period database option.

Understanding Retention Periods

  • By default, Spanner databases retain history for 1 hour.
  • You can configure this retention window to be anywhere from 1 hour up to 7 days.
  • Longer retention periods consume more storage space for keeping older versions of updated or deleted rows.

Set the Retention Option (DDL)

Run the following DDL command to configure test-database to keep historical records for 3 days:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database \
  --instance=test-instance \
  --ddl="ALTER DATABASE \`test-database\` SET OPTIONS (version_retention_period = '3d')"

Verify Database Configuration Settings

Verify that your database configuration options reflect the new 3-day history retention:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases describe test-database \
  --instance=test-instance
  • Configure version retention via DDL Set version_retention_period to '3d' in test-database.
  • Verify metadata settings Inspect databases description to check the retention setting.

Step 3: Simulating a Data Failure

To demonstrate recovery, we will write database records, document the exact timestamp of their existence, and then simulate logical data corruption by dropping all rows.

1. Populate Records

Ensure your TradingAccounts table contains clean test data. Run these commands:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="INSERT INTO TradingAccounts (AccountId, AccountName, FirmName) VALUES (10, 'John', 'Doe_Trading'), (11, 'Jane', 'Smith_Trading')"

2. Note the Current Timestamp

In a real failure scenario, you need to know the timestamp of the database *before* the corruption occurred. Retrieve the current system timestamp. Write down the result (e.g. 2026-07-17T13:05:00.123456Z):

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="SELECT CURRENT_TIMESTAMP()"

3. Simulate Disaster / Deletion

Now, run a destructive query that deletes all rows in the TradingAccounts table (a common user error!):

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="DELETE FROM TradingAccounts WHERE 1=1"

4. Verify Data is Gone

Confirm that the table is now completely empty:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="SELECT * FROM TradingAccounts"
  • Document pre-outage timestamp Run SELECT CURRENT_TIMESTAMP() and note the output time.
  • Run destructive DELETE command Delete all records from TradingAccounts and verify table shows zero rows.

Step 4: Executing Stale Reads

Because you configured a 3-day retention period, the deleted rows are not permanently purged. Spanner still retains the versions that existed at your noted timestamp. You can access this data using **Stale Reads**.

Read-Only Transactions with Timestamps

Spanner allows you to perform a query specifying a past target timestamp. The database executes the query using the snapshot matching that exact time.

Query the Past

Execute the select query, passing the timestamp you recorded in Step 3 into the --read-timestamp flag:

Terminal (Bash / PowerShell)
Copied! 
# Replace the timestamp below with the time you recorded in Step 3
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --read-timestamp="2026-07-17T13:05:00.123456Z" \
  --sql="SELECT * FROM TradingAccounts"
Success: Rows Returned!
Observe that the query returned your deleted trading accounts! The database read the state of the table before the deletion query occurred.
  • Execute query using --read-timestamp Extract historical data from database version records.

Step 5: Restoring the Database (PITR Backup/Restore)

While stale reads are excellent for ad-hoc queries, full disaster recovery requires restoring the entire database to a clean state. We accomplish this by creating a backup from a historical `version-time` and restoring it.

1. Create a Backup from the Past

We will instruct Spanner to create a backup of test-database. Crucially, we supply the --version-time flag, which directs Spanner to take a snapshot of the database state as of our noted timestamp, rather than the current corrupt state:

Terminal (Bash / PowerShell)
Copied! 
# Replace version-time with your recorded pre-outage timestamp
gcloud spanner backups create test-database-backup \
  --instance=test-instance \
  --database=test-database \
  --version-time="2026-07-17T13:05:00.123456Z" \
  --retention-period=3d

2. Restore the Backup to a New Database

Now, restore the backup. We will restore it as a new database named recovered-db:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases restore \
  --destination-database=recovered-db \
  --destination-instance=test-instance \
  --backup=test-database-backup \
  --backup-instance=test-instance

3. Verify Recovery Results

Query your new database to verify that the deleted rows have been successfully recovered:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql recovered-db \
  --instance=test-instance \
  --sql="SELECT * FROM TradingAccounts"
  • Create a point-in-time backup Generate backup using the --version-time flag pointing to the target state.
  • Restore database from backup Deploy backup to 'recovered-db' and query TradingAccounts to verify data restoration.

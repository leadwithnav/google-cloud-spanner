Lab 5: Fine-Grained Access Control (FGAC)
Estimated Duration: 45 mins
Difficulty: Intermediate

Implementing Fine-Grained Access Control (FGAC)

Welcome! In this hands-on lab, you will learn how to secure table resources in Google Cloud Spanner using Fine-Grained Access Control (FGAC) by declaring database roles, column-level select limits, and row-level security (RLS) policies.

Why Do We Need FGAC?

In standard enterprise deployments, Google Cloud IAM manages database access at the resource level (e.g. assigning the roles/spanner.databaseUser role grants full select/write access to all tables in the database). However, modern security architectures demand the **Principle of Least Privilege**, restricting users or client microservices to specific rows and columns.

Spanner supports this using **Fine-Grained Access Control (FGAC)**. With FGAC, you define custom **Database Roles** in the schema. You then grant privileges (like SELECT, INSERT, UPDATE) to these roles for specific tables, views, or subsets of columns. You can also configure **Row-Level Security (RLS)** filters using conditions that restrict query scopes dynamically.

The Scenario
We have a table storing employee records. We will create two database roles:
  • hr_role: Can view all fields and all rows.
  • staff_role: Can only view standard columns (excluding sensitive data like Salary and SSN), and is restricted to viewing rows belonging to their own department (Engineering).
  • Understand FGAC objectives Review the security scenarios for the hr_role and staff_role definitions.

Step 1: Creating the Employees Schema

First, we will deploy the core `Employees` table containing salary and SSN fields, and populate it with sample department records.

1. Deploy the DDL Schema

Deploy the DDL update on your emulator instance:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database \
  --instance=test-instance \
  --ddl="CREATE TABLE Employees (
    EmployeeId INT64 NOT NULL,
    Name STRING(255),
    Email STRING(255),
    Department STRING(100),
    Salary NUMERIC,
    SSN STRING(50)
  ) PRIMARY KEY (EmployeeId);"

2. Insert Sample Records

Add employees belonging to the `Engineering` and `Sales` departments:

Terminal (Bash / PowerShell)

  --instance=test-instance \

  (201, 'Bob Miller', 'bob@company.com', 'Engineering', 95000, 'SSN-999-11'),

Step 2: Defining Database Roles

Database roles are defined directly inside the database schema. Unlike cloud roles, these roles are product-specific and are packaged inside your DDL migrations.

Create Roles

We will create the hr_role and staff_role database roles. Run the DDL command:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database \
  --instance=test-instance \
  --ddl="CREATE ROLE hr_role; CREATE ROLE staff_role;"
  • Create database roles Run DDL statements to initialize 'hr_role' and 'staff_role'.

Step 3: Column-Level Access Control

Column-level access control restricts database roles to querying a specific subset of columns in a table. If a role attempts to query a restricted column, Spanner blocks the request.

Grant Select Permissions

We want to configure the following permissions:

  1. hr_role: Grant read permissions on all columns in the Employees table.
  2. staff_role: Grant read permissions only on non-sensitive columns: EmployeeId, Name, Email, and Department.

Run the DDL command to apply these permissions:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database \
  --instance=test-instance \
  --ddl="GRANT SELECT ON TABLE Employees TO ROLE hr_role; 
         GRANT SELECT (EmployeeId, Name, Email, Department) ON TABLE Employees TO ROLE staff_role;"
  • Apply column-level grants Grant SELECT permissions to hr_role (all columns) and staff_role (specific columns).

Step 4: Row-Level Security (RLS) Policies

Row-Level Security (RLS) allows you to define filter conditions on tables for specific database roles. When users query a table with an active RLS policy, Spanner automatically filters the returned rows to match the policy's conditions.

Create an RLS Policy

We want to restrict staff_role to only see employee records belonging to the Engineering department. We define this policy using a DDL statement:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database \
  --instance=test-instance \
  --ddl="CREATE POLICY engineering_only_policy ON Employees
         FOR SELECT
         TO ROLE staff_role
         USING (Department = 'Engineering');"
Multiple Policies
You can attach multiple RLS policies to a table. Spanner combines policies using logical OR operators. If no RLS policies are defined for a role, that role can access all rows (assuming they have table-level permissions).
  • Deploy the RLS Policy Deploy the CREATE POLICY statement to restrict row visibility for staff_role.

Step 5: Testing and Verifying FGAC Rules

Let's verify that the access control rules are working as expected by querying the database using different active roles.

1. Test hr_role Access

Run a query as `hr_role` by passing the database role using the --database-role flag. The HR role should see all columns and all rows:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --database-role=hr_role \
  --sql="SELECT * FROM Employees"

2. Test staff_role Restricted Column Access

Run a query as `staff_role` requesting all columns: SELECT * FROM Employees. This query should fail because `staff_role` does not have access to the `Salary` or `SSN` columns:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --database-role=staff_role \
  --sql="SELECT * FROM Employees"

Confirm that Spanner blocks the request, returning a permission denied error.

3. Test staff_role Allowed Column & Row Access

Run a query as `staff_role` requesting only the permitted columns. The query should succeed, and the results should be filtered to only show employees in the **Engineering** department (filtering out Sales employees):

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --database-role=staff_role \
  --sql="SELECT EmployeeId, Name, Email, Department FROM Employees"
  • Test the HR role permissions Verify that hr_role can view all fields and all rows.
  • Test the Staff role permissions Verify that staff_role is blocked from viewing sensitive columns, and restricted to Engineering rows.

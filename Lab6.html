Lab 5: Fine-Grained Access Control (FGAC)
Estimated Duration: 75 mins
Difficulty: Intermediate

Implementing Fine-Grained Access Control (FGAC)

Welcome! In this hands-on lab, you will learn how to secure table resources in Google Cloud Spanner using Fine-Grained Access Control (FGAC) by declaring database roles and applying column-level SELECT, INSERT, and UPDATE privileges, revoking access, inheriting roles, and filtering rows through definer-rights views.

Why Do We Need FGAC?

In standard enterprise deployments, Google Cloud IAM manages database access at the resource level (e.g. assigning the roles/spanner.databaseUser role grants full select/write access to all tables in the database). However, modern security architectures demand the **Principle of Least Privilege**, restricting users or client microservices to specific rows and columns.

Spanner supports this using **Fine-Grained Access Control (FGAC)**. With FGAC, you define custom **Database Roles** in the schema. You then grant privileges (like SELECT, INSERT, UPDATE) to these roles for specific tables or subsets of columns — ensuring each role sees only the data it needs.

The Scenario
We have a table storing employee records. We will create two database roles:
  • hr_role: Can view all fields and all rows.
  • staff_role: Can only view standard columns (excluding sensitive data like Salary and SSN).
  • Understand FGAC objectives Review the security scenarios for the hr_role and staff_role definitions.

Step 1: Creating the Employees Schema

First, we will deploy the core `Employees` table containing salary and SSN fields, and populate it with sample department records.

1. Deploy the DDL Schema

Deploy the DDL update on your emulator instance:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database1 \
  --instance=test-instance1 \
  --ddl="CREATE TABLE Employees (
    EmployeeId INT64 NOT NULL,
    Name STRING(255),
    Email STRING(255),
    Department STRING(100),
    Salary NUMERIC,
    SSN STRING(50)
  ) PRIMARY KEY (EmployeeId);"

2. Insert Sample Records

Add employees belonging to the `Engineering` and `Sales` departments:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database1 \
  --instance=test-instance1 \
  --sql="INSERT INTO Employees (EmployeeId, Name, Email, Department, Salary, SSN) VALUES
  (201, 'Bob Miller', 'bob@company.com', 'Engineering', 95000, 'SSN-999-11'),
  (202, 'Alice Croft', 'alice@company.com', 'Engineering', 105000, 'SSN-999-22'),
  (203, 'Charlie Dave', 'charlie@company.com', 'Sales', 80000, 'SSN-999-33')"
  • Deploy Employees table Verify that the target table DDL compiles correctly.
  • Insert sample employees Populate Bobs, Alices, and Charlies into the employee records.

Step 2: Defining Database Roles

Database roles are defined directly inside the database schema. Unlike cloud roles, these roles are product-specific and are packaged inside your DDL migrations.

Create Roles

We will create the hr_role and staff_role database roles. Run the DDL command:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database1 \
  --instance=test-instance1 \
  --ddl="CREATE ROLE hr_role; CREATE ROLE staff_role;"
  • Create database roles Run DDL statements to initialize 'hr_role' and 'staff_role'.

Step 3: Column-Level Access Control

Column-level access control restricts database roles to querying a specific subset of columns in a table. If a role attempts to query a restricted column, Spanner blocks the request.

Grant Select Permissions

We want to configure the following permissions:

  1. hr_role: Grant read permissions on all columns in the Employees table.
  2. staff_role: Grant read permissions only on non-sensitive columns: EmployeeId, Name, Email, and Department.

Run the DDL command to apply these permissions:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database1 \
  --instance=test-instance1 \
  --ddl="GRANT SELECT ON TABLE Employees TO ROLE hr_role; 
         GRANT SELECT (EmployeeId, Name, Email, Department) ON TABLE Employees TO ROLE staff_role;"
  • Apply column-level grants Grant SELECT permissions to hr_role (all columns) and staff_role (specific columns).

Step 4: Testing and Verifying FGAC Rules

Let's verify that the access control rules are working as expected by querying the database using different active roles.

1. Test hr_role Access

Run a query as `hr_role` by passing the database role using the --database-role flag. The HR role should see all columns and all rows:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database1 \
  --instance=test-instance1 \
  --database-role=hr_role \
  --sql="SELECT * FROM Employees"

2. Test staff_role Restricted Column Access

Run a query as `staff_role` requesting all columns: SELECT * FROM Employees. This query should fail because `staff_role` does not have access to the `Salary` or `SSN` columns:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database1 \
  --instance=test-instance1 \
  --database-role=staff_role \
  --sql="SELECT * FROM Employees"

Confirm that Spanner blocks the request, returning a permission denied error.

3. Test staff_role Allowed Column Query

Run a query as `staff_role` requesting only the permitted columns. The query should succeed and return the allowed columns for all employees:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database1 \
  --instance=test-instance1 \
  --database-role=staff_role \
  --sql="SELECT EmployeeId, Name, Email, Department FROM Employees"
  • Test the HR role permissions Verify that hr_role can view all fields and all rows.
  • Test the Staff role permissions Verify that staff_role is blocked from viewing sensitive columns (Salary and SSN).

Step 5: Grant Column-Level INSERT and UPDATE

Spanner supports column-level privileges for SELECT, INSERT, and UPDATE.

1. Grant Column-Level INSERT to hr_role

The omitted Salary and SSN columns are nullable in this lab.

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database1 \
  --instance=test-instance1 \
  --ddl="GRANT INSERT (EmployeeId, Name, Email, Department)
         ON TABLE Employees TO ROLE hr_role;"

2. Grant Column-Level UPDATE to staff_role

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database1 \
  --instance=test-instance1 \
  --ddl="GRANT UPDATE (Email)
         ON TABLE Employees TO ROLE staff_role;"
Predicate columns also matter
The role also needs SELECT access to columns referenced in expressions and WHERE predicates.
  • Grant column-level write privileges Grant INSERT to hr_role and UPDATE(Email) to staff_role.

1. Allowed INSERT as hr_role

         (EmployeeId, Name, Email, Department, Salary)

This should fail because Salary was not included in the INSERT grant.

3. Allowed UPDATE as staff_role

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database1 \
  --instance=test-instance1 \
  --database-role=staff_role \
  --sql="UPDATE Employees
         SET Email='bob.miller@company.com'
         WHERE EmployeeId=201"

4. Disallowed UPDATE of Department

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database1 \
  --instance=test-instance1 \
  --database-role=staff_role \
  --sql="UPDATE Employees
         SET Department='Finance'

DELETE cannot be column-level
DELETE removes the complete row. It can only be granted at table level.
Spanner DDL
Copied! 
-- Invalid
GRANT DELETE (EmployeeId) ON TABLE Employees TO ROLE staff_role;

-- Valid table-level syntax
GRANT DELETE ON TABLE Employees TO ROLE staff_role;
  • Test column-level writes Verify allowed and denied INSERT/UPDATE operations and table-level DELETE behavior.

Terminal (Bash / PowerShell)

Step 8: Role Inheritance

One Spanner database role can inherit privileges from another role.

1. Create the Role Hierarchy

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database1 \
  --instance=test-instance1 \
  --ddl="CREATE ROLE employee_reader_role;
         CREATE ROLE manager_role;
         GRANT SELECT (EmployeeId, Name, Email, Department)
           ON TABLE Employees TO ROLE employee_reader_role;
         GRANT SELECT (Salary)
           ON TABLE Employees TO ROLE manager_role;
         GRANT ROLE employee_reader_role TO ROLE manager_role;"

2. Test Inherited and Direct Privileges

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database1 \
  --instance=test-instance1 \
  --database-role=manager_role \
  --sql="SELECT EmployeeId, Name, Department, Salary
         FROM Employees"

3. Confirm SSN Is Still Restricted

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database1 \
  --instance=test-instance1 \
  --database-role=manager_role \
  --sql="SELECT * FROM Employees"

The final query should fail because neither role has access to SSN.

  • Test role inheritance Confirm manager_role receives inherited and directly granted privileges.

Step 9: Row Filtering Through Views

Spanner does not support PostgreSQL-style CREATE POLICY. Use a definer-rights view to expose selected rows without granting access to the base table.

1. Create a Filtered Definer-Rights View

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database1 \
  --instance=test-instance1 \
  --ddl="CREATE ROLE engineering_role;
         CREATE VIEW EngineeringEmployees
         SQL SECURITY DEFINER AS
         SELECT EmployeeId, Name, Email, Department
         FROM Employees
         WHERE Department = 'Engineering';
         GRANT SELECT ON VIEW EngineeringEmployees
         TO ROLE engineering_role;"

2. Query the Filtered View

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database1 \
  --instance=test-instance1 \
  --database-role=engineering_role \
  --sql="SELECT * FROM EngineeringEmployees"

3. Confirm Base-Table Access Is Denied

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database1 \
  --instance=test-instance1 \
  --database-role=engineering_role \
  --sql="SELECT EmployeeId, Name FROM Employees"
Why SQL SECURITY DEFINER?
An invoker-rights view requires the caller to have privileges on underlying objects. A definer-rights view supports access through the view without direct base-table access.
  • Implement row-filtered access Test allowed Engineering rows and denied direct access to Employees.

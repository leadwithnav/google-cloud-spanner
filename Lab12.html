Lab 12: Change Streams Analytics
Estimated Duration: 45 mins

Real-time Analytics with Spanner Change Streams

Welcome! In this advanced integration lab, you will configure Cloud Spanner Change Streams to capture database modifications (inserts, updates, deletes) in real-time, and route them to Google BigQuery using Cloud Dataflow for sub-second analytical reporting.

The Operational vs Analytical Conflict

Running complex, aggregate analytical queries directly on a highly active OLTP transactional database is a performance anti-pattern. Doing so acquires shared database locks, increases transaction latencies, and threatens system availability.

The solution is **Change Data Capture (CDC)**. By configuring a Change Stream in Cloud Spanner, you stream row changes out immediately, decoupling operational transactions from BigQuery analytics queries.

Key Core Concepts Covered
During this lab, you will configure a Spanner **Change Stream** using DDL, launch a streaming **Dataflow** job, and write transactional history logs into **BigQuery** schemas.

Step 1: Spanner Change Stream Internals

Before launching pipelines, let's learn how Spanner change streams process logs internally.

Under the Hood: Change Stream Architecture

A change stream automatically tracks insertions, updates, and deletions. It does not introduce locking overhead or delay application transactions. Key mechanics include:

  • Retention Period: Captured changes are stored in internal Spanner system metadata tables. You can configure data retention from 1 to 7 days (default is 1 day).
  • Scaling Partitions: Spanner change streams partition automatically to scale with your database split count, allowing concurrent consumers to read stream parts in parallel.

Spanner to BigQuery Dataflow Template

The Google-provided Dataflow template consumes the change stream, decodes the transaction records, and outputs JSON log records containing:

  • mod_type: Insert, Update, or Delete indicator.
  • commit_timestamp: Spanner transaction commit timestamp.
  • row_data: A JSON representation of the new row values.

Step 2: Declaring the Change Stream in Spanner

Let's create the Change Stream using DDL to monitor alterations on the Customers table.

Deploy the Change Stream DDL

Execute the DDL update command to create a change stream. You can watch specific tables or watch the entire database:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database \
  --instance=test-instance \
  --ddl="CREATE CHANGE STREAM CustomerChangeStream FOR Customers;"
Did you know?
You can configure change streams to capture column-level changes to save streaming costs: CREATE CHANGE STREAM EmailChangeStream FOR Customers(Email).
  • Create the change stream Deploy the CustomerChangeStream DDL tracking on test-database.

Step 3: Launching the Change Stream pipeline

Now, deploy the Dataflow job linking Spanner's change log stream to BigQuery.

Dataflow job invocation

Execute the job template pointing to the metadata storage database. Dataflow uses a metadata database to coordinate lock offsets across workers:

Terminal (Bash / PowerShell)
Copied! 
gcloud dataflow jobs run spanner-cdc-to-bigquery \
  --gcs-location-value=gs://dataflow-templates/latest/Spanner_Change_Streams_to_BigQuery \
  --parameters \
spannerInstanceId=test-instance,\
spannerDatabaseId=test-database,\
spannerMetadataInstanceId=test-instance,\
spannerMetadataDatabaseId=test-database,\
spannerChangeStreamName=CustomerChangeStream,\
bigQueryDataset=customer_analytics_dataset
  • Verify Dataflow settings Understand how spannerMetadata properties synchronize offsets among concurrent partition scanners.

Step 4: BigQuery Destination Logs

BigQuery will automatically receive structured change log logs. You can query the log table to audit changes over time.

Analyzing Changes in BigQuery

Unlike Spanner, which only holds the latest row state, BigQuery logs the complete historical modifications stream. You can write analytical audits like: "How many times did customer emails change?"

bigquery_audit.sql
Copied! 
SELECT 
  JSON_VALUE(row_data.CustomerId) as customer_id,
  mod_type,
  commit_timestamp,
  JSON_VALUE(row_data.Email) as current_email
FROM `customer_analytics_dataset.Customers_changelog`
ORDER BY commit_timestamp DESC;
  • Observe log output formats Verify that mod_type (INSERT/UPDATE/DELETE) acts as the operational historical indicator.

Use Case

Analytical queries degrade OLTP performance. Offloading data processing to BigQuery using Change Streams protects Spanner transaction resources.

Pro Tip

Change stream retention defaults to 1 day. If your Dataflow job goes offline, make sure it restarts before the retention window closes to avoid data loss!

Goal

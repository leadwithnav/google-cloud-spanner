Instructor Lab Guide

Lab 4: Banking Schema Refactoring and Index Optimization

This guide mirrors the student lab flow while using a banking example for every schema and query pattern.

Lab Overview

Goal: show how to refactor a normalized banking schema into a Spanner-friendly structure using interleaving and covered indexes.

Audience: learners who are ready to think about data locality and read performance.

Duration: 45–60 minutes.

  • Analyze a legacy banking schema.
  • Refactor tables with interleaving.
  • Add indexes that support account and transaction lookups.
  • Deploy and verify the schema in the emulator.

Step 2: Analyze the Legacy Schema

A traditional normalized design might place customer, account, and transaction records in separate tables. That is fine for transactional systems, but it creates latency when Spanner must read across many splits.

Legacy Schema DDL 
CREATE TABLE Customers (
  CustomerId INT64 NOT NULL,
  Name STRING(255),
  PRIMARY KEY (CustomerId)
);

CREATE TABLE Accounts (
  AccountId INT64 NOT NULL,
  CustomerId INT64 NOT NULL,
  AccountType STRING(32),
  Balance NUMERIC,
  PRIMARY KEY (AccountId)
);

CREATE TABLE Transactions (
  TransactionId INT64 NOT NULL,
  AccountId INT64 NOT NULL,
  TransactionDate DATE,
  Amount NUMERIC,
  Status STRING(32),
  PRIMARY KEY (TransactionId)
);
Instructor focus: point out that joins between customers, accounts, and transactions are expensive at scale and should be co-located.

Step 3: Refactor with Interleaving

Make the schema hierarchical: each customer has accounts, and each account has transactions. The child tables inherit the parent key prefix.

Interleaved Schema DDL 
CREATE TABLE Customers (
  CustomerId INT64 NOT NULL,
  Name STRING(255),
  PRIMARY KEY (CustomerId)
);

CREATE TABLE Accounts (
  CustomerId INT64 NOT NULL,
  AccountId INT64 NOT NULL,
  AccountType STRING(32),
  Balance NUMERIC,
  PRIMARY KEY (CustomerId, AccountId)
) INTERLEAVE IN PARENT Customers ON DELETE CASCADE;

CREATE TABLE Transactions (
  CustomerId INT64 NOT NULL,
  AccountId INT64 NOT NULL,
  TransactionId INT64 NOT NULL,
  TransactionDate DATE,
  Amount NUMERIC,
  Status STRING(32),
  PRIMARY KEY (CustomerId, AccountId, TransactionId)
) INTERLEAVE IN PARENT Accounts ON DELETE CASCADE;
Instructor focus: explain that the child key must start with the parent key prefix for interleaving to work.

Step 4: Optimize Indexes

To query transactions by date, use a secondary index. To reduce back-joins, store commonly requested columns directly inside the index.

Index DDL 
CREATE INDEX TransactionsByDate
ON Transactions(TransactionDate);

CREATE INDEX TransactionsByDateCovered
ON Transactions(TransactionDate)
STORING (Amount, Status);
Instructor focus: demonstrate that STORING removes the need for an extra read of the base table for common columns.

Step 5: Deploy the Schema

Write the DDL into a schema file and apply it with the emulator using the gcloud CLI.

Terminal 
gcloud spanner databases ddl update test-database \
  --instance=multi-region-spanner \
  --ddl-file=schema.ddl
Instructor focus: make sure learners can inspect the metadata and see that tables and indexes are registered properly.

Step 6: Checkout Plans

Insert sample account and transaction rows, then query them to show how locality improves reads.

Terminal 
gcloud spanner databases execute-sql test-database \
  --instance=multi-region-spanner \
  --sql="INSERT INTO Customers (CustomerId, Name) VALUES (501, 'Ava Patel')"

gcloud spanner databases execute-sql test-database \
  --instance=multi-region-spanner \
  --sql="INSERT INTO Accounts (CustomerId, AccountId, AccountType, Balance) VALUES (501, 1001, 'Savings', 15000.00)"

gcloud spanner databases execute-sql test-database \
  --instance=multi-region-spanner \
  --sql="INSERT INTO Transactions (CustomerId, AccountId, TransactionId, TransactionDate, Amount, Status) VALUES (501, 1001, 9001, '2026-07-17', 1250.00, 'Posted')"

gcloud spanner databases execute-sql test-database \
  --instance=multi-region-spanner \
  --sql="SELECT CustomerId, AccountId, TransactionId, Amount FROM Transactions@{FORCE_INDEX=TransactionsByDateCovered} WHERE TransactionDate = '2026-07-17'"
Instructor focus: connect the query result to the fact that the index can serve the request without extra reads for the stored columns.

Discussion 1: Who Owns the Schema?

Scenario: The banking application is becoming slow when displaying account statements.

DBA Perspective
  • Existing schema causes expensive joins.
  • Missing indexes.
  • Need to analyze query plans.
  • Identify hotspots.
  • Optimize storage layout.
Application Engineer Perspective
  • Designed tables without considering access patterns.
  • Fetches customer → account → transaction separately.
  • Makes multiple network calls.
  • ORM generates inefficient SQL.

Discussion 2: Normalize Everything?

Which is more important?

  • Perfect normalization?
  • Read performance?
DBA Perspective
  • Normalization avoids redundancy.
Application Engineer Perspective
  • My screen always loads account + transactions together.
  • Spanner favors access pattern driven schema design.
  • Schema should reflect how the application actually reads data.

Discussion 3: When NOT to Interleave?

Good Candidates
  • Orders → OrderItems
  • Account → Transactions
  • Customer → Addresses
Poor Candidates
  • Audit logs
  • Login history
  • Marketing campaigns
Interleave when: parent and child are almost always accessed together.
Avoid when: child grows independently, has different access patterns, or has a different lifecycle.

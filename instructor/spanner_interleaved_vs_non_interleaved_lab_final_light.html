Interleaved vs Non-Interleaved Tables

Create equivalent schemas, load identical data using mutation-safe batches, run equivalent joins, and compare their Cloud Spanner execution plans.

1. Lab Objective

Interleaved design

CustomersInterleaved → OrdersInterleaved

Regular design

CustomersRegular + OrdersRegular

Both designs use identical columns, primary keys, values, and row counts. The only intended difference is physical parent-child interleaving.

Run the plan comparison on a managed Cloud Spanner instance. The emulator is primarily intended for functional testing.

2. Create the Tables

2.1 Interleaved parent

01_customers_interleaved.sqlSQL
CREATE TABLE CustomersInterleaved (
  CustomerId INT64 NOT NULL,
  CustomerName STRING(100) NOT NULL,
  CustomerType STRING(20),
  City STRING(50),
) PRIMARY KEY (CustomerId);

2.2 Interleaved child

02_orders_interleaved.sqlSQL
CREATE TABLE OrdersInterleaved (
  CustomerId INT64 NOT NULL,
  OrderId INT64 NOT NULL,
  OrderDate DATE NOT NULL,
  Amount NUMERIC NOT NULL,
  OrderStatus STRING(20),
) PRIMARY KEY (CustomerId, OrderId),
  INTERLEAVE IN PARENT CustomersInterleaved
  ON DELETE CASCADE;

2.3 Regular tables

03_regular_tables.sqlSQL
CREATE TABLE CustomersRegular (
  CustomerId INT64 NOT NULL,
  CustomerName STRING(100) NOT NULL,
  CustomerType STRING(20),
  City STRING(50),
) PRIMARY KEY (CustomerId);

CREATE TABLE OrdersRegular (
  CustomerId INT64 NOT NULL,
  OrderId INT64 NOT NULL,
  OrderDate DATE NOT NULL,
  Amount NUMERIC NOT NULL,
  OrderStatus STRING(20),
) PRIMARY KEY (CustomerId, OrderId);

5. Why the Original Order Insert Failed

Error: The transaction contains too many mutations. Maximum: 80,000.

The original statement attempted to insert:

1,000 customers × 20 orders = 20,000 rows

Each row writes five columns, so the approximate mutation count is:

20,000 × 5 = 100,000 mutations

The corrected lab uses four separate batches of 250 customers. Each batch inserts 5,000 order rows, or approximately 25,000 base-table mutations before considering indexes.
Run each batch as a separate statement. Do not wrap all four statements in one explicit read-write transaction.

6. Insert Interleaved Orders in Four Batches

Run each statement separately.

Batch 1: Customers 1–250

interleaved_orders_batch_1.sqlSQL
INSERT INTO OrdersInterleaved (
  CustomerId, OrderId, OrderDate, Amount, OrderStatus
)
SELECT
  CustomerId,
  OrderNumber,
  DATE_ADD(
    DATE '2026-01-01',
    INTERVAL MOD(CustomerId + OrderNumber, 180) DAY
  ),
  CAST(
    100 + MOD((CustomerId * 17) + (OrderNumber * 31), 5000)
    AS NUMERIC
  ),
  CASE MOD(OrderNumber, 4)
    WHEN 0 THEN 'PENDING'
    WHEN 1 THEN 'COMPLETED'
    WHEN 2 THEN 'SHIPPED'
    ELSE 'CANCELLED'
  END
FROM UNNEST(GENERATE_ARRAY(1, 250)) AS CustomerId
CROSS JOIN UNNEST(GENERATE_ARRAY(1, 20)) AS OrderNumber;

9. Join Experiments

9.1 One customer

join_one_customer.sqlSQL
-- Interleaved
SELECT c.CustomerId, c.CustomerName, o.OrderId, o.OrderDate, o.Amount
FROM CustomersInterleaved AS c
JOIN OrdersInterleaved AS o
  ON c.CustomerId = o.CustomerId
WHERE c.CustomerId = 500;

-- Non-interleaved
SELECT c.CustomerId, c.CustomerName, o.OrderId, o.OrderDate, o.Amount
FROM CustomersRegular AS c
JOIN OrdersRegular AS o
  ON c.CustomerId = o.CustomerId
WHERE c.CustomerId = 500;

9.2 Customer range

join_customer_range.sqlSQL
-- Interleaved
SELECT c.CustomerId, c.CustomerName, o.OrderId, o.Amount
FROM CustomersInterleaved AS c
JOIN OrdersInterleaved AS o
  ON c.CustomerId = o.CustomerId
WHERE c.CustomerId BETWEEN 400 AND 500;

-- Non-interleaved
SELECT c.CustomerId, c.CustomerName, o.OrderId, o.Amount
FROM CustomersRegular AS c
JOIN OrdersRegular AS o
  ON c.CustomerId = o.CustomerId
WHERE c.CustomerId BETWEEN 400 AND 500;

9.3 Full join with aggregation

full_join_aggregation.sqlSQL
-- Interleaved
SELECT c.CustomerType, COUNT(*) AS OrderCount, SUM(o.Amount) AS TotalAmount
FROM CustomersInterleaved AS c
JOIN OrdersInterleaved AS o
  ON c.CustomerId = o.CustomerId
GROUP BY c.CustomerType;

-- Non-interleaved
SELECT c.CustomerType, COUNT(*) AS OrderCount, SUM(o.Amount) AS TotalAmount
FROM CustomersRegular AS c
JOIN OrdersRegular AS o
  ON c.CustomerId = o.CustomerId
GROUP BY c.CustomerType;

9.4 Filter on a non-key column

filter_order_status.sqlSQL
-- Interleaved
SELECT c.CustomerId, c.CustomerName, o.OrderId, o.Amount
FROM CustomersInterleaved AS c
JOIN OrdersInterleaved AS o
  ON c.CustomerId = o.CustomerId
WHERE o.OrderStatus = 'PENDING';

-- Non-interleaved
SELECT c.CustomerId, c.CustomerName, o.OrderId, o.Amount
FROM CustomersRegular AS c
JOIN OrdersRegular AS o
  ON c.CustomerId = o.CustomerId
WHERE o.OrderStatus = 'PENDING';

10. Optional Covered-Index Experiment

create_covered_indexes.sqlSQL
CREATE INDEX OrdersInterleavedByStatus
ON OrdersInterleaved (OrderStatus, CustomerId, OrderId)
STORING (Amount, OrderDate);

CREATE INDEX OrdersRegularByStatus
ON OrdersRegular (OrderStatus, CustomerId, OrderId)
STORING (Amount, OrderDate);
Create the indexes after loading the initial data. If indexes already exist during inserts, index maintenance adds more mutations to each transaction.

11. View and Compare the Execution Plan

  1. Open Google Cloud Console and go to Spanner.
  2. Select the instance and database.
  3. Open Spanner Studio.
  4. Paste one query at a time.
  5. Choose Plan to inspect the optimizer plan.
  6. Choose Profile to execute and collect runtime statistics.
  7. Inspect scans, key ranges, join operators, distributed operators, rows read, CPU time, and elapsed time.
Do not expect interleaving to guarantee one specific operator. The optimizer may choose Apply, Distributed Apply, Hash Join, Merge Join, or another strategy.

13. Final Conclusion

Interleaving improves locality for parent-child access through the parent key. It does not replace secondary indexes for unrelated filters and does not guarantee that a query avoids distributed execution.

Instructor takeaway: Interleaving changes physical locality; the execution plan reveals how the cost-based optimizer actually uses that locality.

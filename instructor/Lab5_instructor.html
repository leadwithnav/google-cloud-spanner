Interleaved vs Non-Interleaved Tables

Create equivalent schemas, load identical datasets, run the same joins, and compare execution plans to understand data locality and join behavior in Cloud Spanner.

Level: Intermediate Estimated Time: 45–60 minutes Platform: Cloud Spanner Focus: Execution Plans

Lab Navigation

1. Objective 2. Interleaved tables 3. Non-interleaved tables 4. Verify schema 5. Load identical data 6. Validate data 7. Single-customer join 8. Range join 9. Full join 10. Non-key filter 11. Optional index test 12. View execution plan 13. Comparison worksheet 14. Operator guide 15. Conclusion

1. Lab Objective

This lab creates two equivalent customer-order schemas:

Interleaved design

CustomersInterleaved → OrdersInterleaved

Regular design

CustomersRegular + OrdersRegular

Both schemas use the same columns, primary keys, row counts, and values. The only intentional difference is the parent-child interleaving relationship.

Run this lab on a real Cloud Spanner instance. The emulator is useful for functional testing, but it does not provide the same execution-plan and runtime profiling experience as the managed service.

Learning outcomes

  • Create interleaved and non-interleaved tables.
  • Insert identical datasets into both designs.
  • Run equivalent joins.
  • Compare scans, distributed operators, join strategies, rows read, CPU time, and latency.
  • Explain when interleaving helps and when an index is still required.

2. Create the Interleaved Tables

2.1 Parent table

CREATE TABLE CustomersInterleaved (
  CustomerId INT64 NOT NULL,
  CustomerName STRING(100) NOT NULL,
  CustomerType STRING(20),
  City STRING(50),
) PRIMARY KEY (CustomerId);

2.2 Interleaved child table

CREATE TABLE OrdersInterleaved (
  CustomerId INT64 NOT NULL,
  OrderId INT64 NOT NULL,
  OrderDate DATE NOT NULL,
  Amount NUMERIC NOT NULL,
  OrderStatus STRING(20),
) PRIMARY KEY (CustomerId, OrderId),
  INTERLEAVE IN PARENT CustomersInterleaved
  ON DELETE CASCADE;
The child primary key starts with the complete parent primary key. This is required for parent-child interleaving. 
Parent key: (CustomerId)
Child key:  (CustomerId, OrderId)

3. Create the Non-Interleaved Tables

3.1 Regular parent table

CREATE TABLE CustomersRegular (
  CustomerId INT64 NOT NULL,
  CustomerName STRING(100) NOT NULL,
  CustomerType STRING(20),
  City STRING(50),
) PRIMARY KEY (CustomerId);

3.2 Regular orders table

CREATE TABLE OrdersRegular (
  CustomerId INT64 NOT NULL,
  OrderId INT64 NOT NULL,
  OrderDate DATE NOT NULL,
  Amount NUMERIC NOT NULL,
  OrderStatus STRING(20),
) PRIMARY KEY (CustomerId, OrderId);
Keep the primary keys identical in both designs. This isolates physical locality as the main variable instead of changing the logical key design.

5. Insert Identical Data

The following example creates 1,000 customers and 20 orders per customer in each design.

5.1 Insert customers into the interleaved parent table

INSERT INTO CustomersInterleaved (
  CustomerId,
  CustomerName,
  CustomerType,
  City
)
SELECT
  CustomerId,
  CONCAT('Customer-', CAST(CustomerId AS STRING)),
  CASE
    WHEN MOD(CustomerId, 3) = 0 THEN 'Premium'
    ELSE 'Standard'
  END,
  CASE MOD(CustomerId, 5)
    WHEN 0 THEN 'Delhi'
    WHEN 1 THEN 'Mumbai'
    WHEN 2 THEN 'Bengaluru'
    WHEN 3 THEN 'London'
    ELSE 'Singapore'
  END
FROM UNNEST(GENERATE_ARRAY(1, 1000)) AS CustomerId;

5.3 Insert orders into the interleaved child table

INSERT INTO OrdersInterleaved (
  CustomerId,
  OrderId,
  OrderDate,
  Amount,
  OrderStatus
)
SELECT
  CustomerId,
  OrderNumber,
  DATE_ADD(
    DATE '2026-01-01',
    INTERVAL MOD(CustomerId + OrderNumber, 180) DAY
  ),
  CAST(
    100 + MOD((CustomerId * 17) + (OrderNumber * 31), 5000)
    AS NUMERIC
  ),
  CASE MOD(OrderNumber, 4)
    WHEN 0 THEN 'PENDING'
    WHEN 1 THEN 'COMPLETED'
    WHEN 2 THEN 'SHIPPED'
    ELSE 'CANCELLED'
  END
FROM UNNEST(GENERATE_ARRAY(1, 1000)) AS CustomerId
CROSS JOIN UNNEST(GENERATE_ARRAY(1, 20)) AS OrderNumber;

7. Experiment 1: Join for One Customer

Interleaved query

SELECT
  c.CustomerId,
  c.CustomerName,
  o.OrderId,
  o.OrderDate,
  o.Amount,
  o.OrderStatus
FROM CustomersInterleaved AS c
JOIN OrdersInterleaved AS o
  ON c.CustomerId = o.CustomerId
WHERE c.CustomerId = 500;

Non-interleaved query

SELECT
  c.CustomerId,
  c.CustomerName,
  o.OrderId,
  o.OrderDate,
  o.Amount,
  o.OrderStatus
FROM CustomersRegular AS c
JOIN OrdersRegular AS o
  ON c.CustomerId = o.CustomerId
WHERE c.CustomerId = 500;

Observe

  • Parent and child scan types.
  • Key ranges used.
  • Join operator selected.
  • Presence of distributed operators.
  • Rows read versus rows returned.
  • CPU time and elapsed time.

8. Experiment 2: Join a Customer Range

Interleaved query

SELECT
  c.CustomerId,
  c.CustomerName,
  o.OrderId,
  o.Amount
FROM CustomersInterleaved AS c
JOIN OrdersInterleaved AS o
  ON c.CustomerId = o.CustomerId
WHERE c.CustomerId BETWEEN 400 AND 500;

Non-interleaved query

SELECT
  c.CustomerId,
  c.CustomerName,
  o.OrderId,
  o.Amount
FROM CustomersRegular AS c
JOIN OrdersRegular AS o
  ON c.CustomerId = o.CustomerId
WHERE c.CustomerId BETWEEN 400 AND 500;

9. Experiment 3: Join All Customers and Orders

Interleaved full join

SELECT
  c.CustomerType,
  COUNT(*) AS OrderCount,
  SUM(o.Amount) AS TotalAmount
FROM CustomersInterleaved AS c
JOIN OrdersInterleaved AS o
  ON c.CustomerId = o.CustomerId
GROUP BY c.CustomerType;

Non-interleaved full join

SELECT
  c.CustomerType,
  COUNT(*) AS OrderCount,
  SUM(o.Amount) AS TotalAmount
FROM CustomersRegular AS c
JOIN OrdersRegular AS o
  ON c.CustomerId = o.CustomerId
GROUP BY c.CustomerType;
For larger inputs, the optimizer may choose Hash Join, Merge Join, Apply, or Distributed Apply. Interleaving does not force one specific operator.

10. Experiment 4: Filter on a Non-Key Column

Interleaved query

SELECT
  c.CustomerId,
  c.CustomerName,
  o.OrderId,
  o.Amount
FROM CustomersInterleaved AS c
JOIN OrdersInterleaved AS o
  ON c.CustomerId = o.CustomerId
WHERE o.OrderStatus = 'PENDING';

Regular query

SELECT
  c.CustomerId,
  c.CustomerName,
  o.OrderId,
  o.Amount
FROM CustomersRegular AS c
JOIN OrdersRegular AS o
  ON c.CustomerId = o.CustomerId
WHERE o.OrderStatus = 'PENDING';
Interleaving organizes data around CustomerId. It does not organize all rows globally by OrderStatus. This query may still require broad scans.

11. Optional Secondary-Index Experiment

Create equivalent covered indexes

CREATE INDEX OrdersInterleavedByStatus
ON OrdersInterleaved (
  OrderStatus,
  CustomerId,
  OrderId
)
STORING (
  Amount,
  OrderDate
);
CREATE INDEX OrdersRegularByStatus
ON OrdersRegular (
  OrderStatus,
  CustomerId,
  OrderId
)
STORING (
  Amount,
  OrderDate
);

Force the index for the interleaved query

SELECT
  c.CustomerId,
  c.CustomerName,
  o.OrderId,
  o.Amount
FROM CustomersInterleaved AS c
JOIN OrdersInterleaved@{FORCE_INDEX=OrdersInterleavedByStatus} AS o
  ON c.CustomerId = o.CustomerId
WHERE o.OrderStatus = 'PENDING';

Force the index for the regular query

SELECT
  c.CustomerId,
  c.CustomerName,
  o.OrderId,
  o.Amount
FROM CustomersRegular AS c
JOIN OrdersRegular@{FORCE_INDEX=OrdersRegularByStatus} AS o
  ON c.CustomerId = o.CustomerId
WHERE o.OrderStatus = 'PENDING';

12. View the Execution Plan

  2. Go to Spanner.
  3. Select the instance and database.
  4. Open Spanner Studio or the query editor.
  5. Paste one of the lab queries.

14. Operator Guide

Table Scan

Reads rows from a base table. Check whether it is a full scan or a key-range scan and whether filters are pushed down.

Apply or Cross Apply

For each row from one side, Spanner performs a lookup on the other side using the join key. This is often effective for selective parent-child access.

Distributed Apply

The Apply operation is distributed across multiple Spanner servers or key ranges.

Hash Join

Spanner builds or uses a hash representation of join keys. It may be efficient when joining larger input sets.

Merge Join

Consumes both inputs in join-key order. Because both designs use (CustomerId, OrderId), either design may be eligible for an efficient ordered join.

Distributed Union

Represents parallel work over multiple splits or remote subplans whose results are combined.

Do not assume “interleaved means no distributed operators.” A large interleaved hierarchy can still span several splits and servers.

15. Final Conclusion

Interleaving is a physical data-locality optimization for parent-child access. It is most helpful when applications retrieve child rows through the parent primary key.

It does not replace indexes for unrelated filters, and it does not guarantee a specific join operator. Cloud Spanner’s cost-based optimizer still chooses the final execution strategy based on filters, statistics, cardinality, ordering, and distribution.

Best teaching takeaway: interleaving improves locality, while the execution plan shows how Spanner actually chose to execute the query.

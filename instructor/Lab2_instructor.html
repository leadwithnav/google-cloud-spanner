Instructor Lab Guide

Lab 2: Multi-Region Deployments, PITR, and Recovery Workflows

This guide mirrors the student lab flow but adds instructor notes, timing guidance, and focus points for teaching the recovery concepts clearly.

Lab Overview

Goal: teach learners how Spanner supports high availability, historical recovery, and point-in-time recovery with a multi-region setup.

Audience: intermediate learners who already know the basics of Spanner DDL/DML.

Duration: 45–60 minutes.

  • Deploy a multi-region Spanner instance.
  • Configure version retention for PITR.
  • Simulate logical failure by deleting rows.
  • Recover data using stale reads and database restore.

Instructor Preparation

  • Confirm the learner has a live Google Cloud project and billing-enabled account.
  • Have the gcloud CLI and Python SDK ready before the session.
  • Explain that multi-region deployments create real costs and should be cleaned up afterwards.
  • Pause if credentials or project selection fail; do not move on until the environment is connected.

Step 1: Introduce the Recovery Scenario

Open by distinguishing between two classes of incidents:

  • Infrastructure failure: a zone or region becomes unavailable. Spanner’s replicas protect availability.
  • Logical failure: bad data is written, a table is deleted, or records are corrupted. PITR is the recovery tool.

Use a simple banking story: a support engineer accidentally deletes the latest account rows during a maintenance window, and the operations team needs to recover balances for customer accounts quickly while preserving audit readiness.

Teaching tip: spend a few minutes on the difference between resilience and recovery; many learners confuse the two concepts.

Step 2: Deploy a Multi-Region Instance

Walk learners through finding the regional configuration and creating the instance. Make clear that this is a production-style activity and incurs billing.

Terminal 
gcloud spanner instance-configs list --filter="name:nam"

gcloud spanner instances create multi-region-spanner \
  --config=nam3 \
  --description="HA Multi-Region Instance" \
  --processing-units=100
Terminal 
gcloud spanner databases create test-database --instance=multi-region-spanner

gcloud spanner databases ddl update test-database \
  --instance=multi-region-spanner \
  --ddl="CREATE TABLE BankAccounts (AccountId INT64 NOT NULL, CustomerId INT64 NOT NULL, AccountType STRING(32), Balance NUMERIC) PRIMARY KEY (AccountId)"
Instructor focus: point out that the instance is being created in a multi-region config, but the lab still uses a simple table so the recovery story stays clear.

Step 3: Configure PITR Retention

Explain that PITR needs historical versions to exist before recovery can be performed. The retention window can be set from 1 hour up to 7 days.

Terminal 
gcloud spanner databases ddl update test-database \
  --instance=multi-region-spanner \
  --ddl="ALTER DATABASE \`test-database\` SET OPTIONS (version_retention_period = '3d')"

gcloud spanner databases describe test-database --instance=multi-region-spanner
Instructor focus: explain that longer retention increases storage cost, so teams pick a reasonable window based on recovery needs.

Step 4: Simulate Data Failure

Keep this step theatrical but controlled. The goal is to show how a logical failure can wipe or corrupt banking records and leave the operations team with a recovery problem.

Terminal 
gcloud spanner databases execute-sql test-database \
  --instance=multi-region-spanner \
  --sql="INSERT INTO BankAccounts (AccountId, CustomerId, AccountType, Balance) VALUES (1001, 501, 'Savings', 15000.00), (1002, 502, 'Checking', 8500.50)"

gcloud spanner databases execute-sql test-database \
  --instance=multi-region-spanner \
  --sql="SELECT CURRENT_TIMESTAMP()"

gcloud spanner databases execute-sql test-database \
  --instance=multi-region-spanner \
  --sql="DELETE FROM BankAccounts WHERE 1=1"

gcloud spanner databases execute-sql test-database \
  --instance=multi-region-spanner \
  --sql="SELECT * FROM BankAccounts"
Instructor focus: point out that the table is empty after the delete, which mirrors a common accidental data loss mistake.

Step 5: Recover with Stale Reads

Show that the historical rows still exist inside Spanner’s retained versions. Stale reads let you query the past without restoring the whole database.

Terminal 
gcloud spanner databases execute-sql test-database \
  --instance=multi-region-spanner \
  --read-timestamp="2026-07-17T13:05:00.123456Z" \
  --sql="SELECT * FROM BankAccounts"
Instructor focus: explain that stale reads are great for investigation and targeted recovery, but they are not the same as full restoration.

Step 6: Restore in Place with Python

Use the Python sample to repopulate the active table by replaying the historical rows into the current table.

Python 
import datetime
from google.cloud import spanner

INSTANCE_ID = "multi-region-spanner"
DATABASE_ID = "test-database"
TIMESTAMP_STR = "2026-07-17T13:05:00.123456Z"

client = spanner.Client()
instance = client.instance(INSTANCE_ID)
database = instance.database(DATABASE_ID)
recovery_time = datetime.datetime.fromisoformat(TIMESTAMP_STR.replace("Z", "+00:00"))

with database.snapshot(read_timestamp=recovery_time) as snapshot:
    rows = list(snapshot.execute_sql("SELECT AccountId, CustomerId, AccountType, Balance FROM BankAccounts"))

if rows:
    database.run_in_transaction(lambda tx, rows=rows: tx.insert_or_update(
        table="BankAccounts",
        columns=["AccountId", "CustomerId", "AccountType", "Balance"],
        values=rows
    ))
    print("Bank account restoration completed")
Terminal 
python Lab2/restore.py
Instructor focus: this is a good moment to explain that in-place restore is often a practical operational response for targeted data recovery.

Step 7: Restore the Database from Backup

For full disaster recovery, restore the database from a backup that was created using the historical version time.

Terminal 
gcloud spanner backups create test-database-backup \
  --instance=multi-region-spanner \
  --database=test-database \
  --version-time="2026-07-17T13:05:00.123456Z" \
  --retention-period=3d --async

gcloud spanner backups describe test-database-backup --instance=multi-region-spanner

gcloud spanner databases restore \
  --destination-database=recovered-db \
  --destination-instance=multi-region-spanner \
  --backup=test-database-backup \
  --backup-instance=multi-region-spanner
Instructor focus: compare this to stale reads and point out that backup/restore is the more complete enterprise recovery path.

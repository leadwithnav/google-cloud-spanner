Instructor Lab Guide

Lab 3: Views and Remote UDFs for Banking Workloads

This guide mirrors the student lab sequence while using a banking scenario for every example, so instructors can teach the same concepts with a more relevant story.

Lab Overview

Goal: show how Cloud Spanner can combine remote serverless logic with SQL views to present banking data in a consumer-friendly way.

Audience: learners who already know basic Spanner DDL and DML.

Duration: 45–60 minutes.

  • Deploy a Cloud Function for balance formatting.
  • Register the function as a Remote UDF in Spanner.
  • Query banking balances through the remote logic.
  • Wrap the logic in a reusable database view.

Instructor Preparation

  • Confirm that the learner has a project with billing enabled.
  • Make sure the Cloud Functions and gcloud CLI are available.
  • Remind learners that this lab uses serverless execution and can create costs if left running.
  • Pause early if the deployment endpoint is not returned correctly.

Step 2: Deploy the Cloud Function

Use a serverless function that formats a numeric balance into US currency. This keeps the presentation logic outside the database and makes it reusable.

Node.js 
const functions = require('@google-cloud/functions-framework');

functions.http('formatAccountBalance', (req, res) => {
  const requestBody = req.body;
  const calls = requestBody.calls || [];

  const replies = calls.map(args => {
    const amount = parseFloat(args[0]);
    if (isNaN(amount)) return null;
    return new Intl.NumberFormat('en-US', { style: 'currency', currency: 'USD' }).format(amount);
  });

  res.status(200).json({ replies });
});
Terminal 
gcloud functions deploy formatAccountBalance \
  --runtime=nodejs20 \
  --trigger-http \
  --allow-unauthenticated \
  --entry-point=formatAccountBalance \
  --region=us-central1
Instructor focus: show that the function is the presentation layer, not the data source.

Step 3: Register the Remote UDF

Register the deployed function as a Remote UDF inside your Spanner database. The endpoint is the bridge between SQL and serverless execution.

DDL SQL 
CREATE SCHEMA udf;

CREATE OR REPLACE FUNCTION udf.formatAccountBalance (amount NUMERIC) RETURNS STRING
  NOT DETERMINISTIC
  LANGUAGE REMOTE
  OPTIONS (
    endpoint = 'https://us-central1-[PROJECT_ID].cloudfunctions.net/formatAccountBalance'
  );
Terminal 
gcloud spanner databases ddl update test-database \
  --instance=multi-region-spanner \
  --ddl="CREATE SCHEMA udf" \
  --ddl="CREATE OR REPLACE FUNCTION udf.formatAccountBalance (amount NUMERIC) \
    RETURNS STRING \
    NOT DETERMINISTIC \
    LANGUAGE REMOTE \
    OPTIONS ( \
      endpoint = 'https://us-central1-[PROJECT_ID].cloudfunctions.net/formatAccountBalance' \
    )"
Instructor focus: explain that the function name and endpoint are both part of the contract between Spanner and the serverless service.

Step 4: Query Using the Remote UDF

Once the function is in place, run SQL that calls it directly. The query remains simple while the formatting logic comes from the cloud function.

Terminal 
gcloud spanner databases ddl update test-database \
  --instance=multi-region-spanner \
  --ddl="ALTER TABLE BankAccounts ADD COLUMN Balance NUMERIC"

gcloud spanner databases execute-sql test-database \
  --instance=multi-region-spanner \
  --sql="UPDATE BankAccounts SET Balance = 12500.75 WHERE AccountId = 1001"

gcloud spanner databases execute-sql test-database \
  --instance=multi-region-spanner \
  --sql="UPDATE BankAccounts SET Balance = 945000.50 WHERE AccountId = 1002"

gcloud spanner databases execute-sql test-database \
  --instance=multi-region-spanner \
  --sql="SELECT AccountId, CustomerId, udf.formatAccountBalance(Balance) AS FormattedBalance FROM BankAccounts"
Instructor focus: highlight that the SQL query stays readable while the formatting work is delegated to the remote service.

Step 5: Create a Database View

Wrap the formatted account query in a view so downstream applications can query a clean semantic interface.

Terminal 
gcloud spanner databases ddl update test-database \
  --instance=multi-region-spanner \
  --ddl="CREATE OR REPLACE VIEW AccountPortfolioView \
    SQL SECURITY INVOKER \
    AS SELECT \
      BankAccounts.AccountId, \
      BankAccounts.CustomerId, \
      BankAccounts.AccountType, \
      udf.formatAccountBalance(BankAccounts.Balance) AS FormattedBalance \
    FROM BankAccounts"

gcloud spanner databases execute-sql test-database \
  --instance=multi-region-spanner \
  --sql="SELECT * FROM AccountPortfolioView"
Instructor focus: explain that the view offers a stable contract to client apps while the implementation details remain flexible.

Step 6: Challenge Extension

Encourage learners to extend the pattern for a banking compliance use case such as high-value account monitoring.

Terminal 
gcloud spanner databases ddl update test-database \
  --instance=multi-region-spanner \
  --ddl="CREATE OR REPLACE VIEW HighValueAccountView \
    SQL SECURITY INVOKER \
    AS SELECT \
      BankAccounts.AccountId, \
      BankAccounts.CustomerId, \
      BankAccounts.AccountType, \
      udf.formatAccountBalance(BankAccounts.Balance) AS FormattedBalance \
    FROM BankAccounts \
    WHERE BankAccounts.Balance >= 100000"
Instructor focus: propose how this could be used for monitoring, fraud review, or client relationship reporting.

Step 7: Wrap-Up Discussion

  • Why might a bank prefer a view over exposing the raw account table directly?
  • When would a remote UDF be better than doing the formatting in the application tier?
  • How would this pattern help with dashboards, statements, or compliance reporting?

Step 8: Cleanup and Final Notes

Scale the Spanner instance back down and remind learners that serverless functions and remote UDFs are powerful but should be used thoughtfully in production.

Terminal 
gcloud spanner instances update multi-region-spanner \
  --processing-units=100

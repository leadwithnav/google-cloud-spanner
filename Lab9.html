Lab 8: Application Integration with Spanner

Learn how to connect console applications to a Google Cloud Spanner instance and perform programmatic reads, writes, and transaction routines.

Introduction

In preceding labs, you provisioned a Cloud Spanner instance and database, updated DDL schemas, and analyzed queries using command-line tools. In this hands-on lab, you will transition to writing application code using the official client libraries.

Google Cloud Spanner client SDKs handle the complex aspects of cloud databases for you, including: session pool management, high-performance gRPC connection pooling, automated query routing to splits, transaction retry logic for aborted operations, and emulator detection.

Core Objectives

1. Local Emulator Routing
Configure the application SDK client to bypass real GCP authentication and route all gRPC calls directly to the local emulator container.
2. Mutations vs. DML
Understand when to utilize the high-speed Mutation API for database edits versus traditional SQL DML commands.
3. Read-Write Transactions
Write robust transactions that coordinate read dependencies and perform consistent modifications while automatically handling transaction aborts.
4. Language Options
Work in the language of your choice: Python, Go, or Node.js. Select your SDK and toggle step instructions interactively.

Step 3: Programmatic Writes using Mutations

In Cloud Spanner, there are two primary APIs for modifying database state: Mutations and DML (Data Manipulation Language) statements.

Deep Dive: Mutations vs DML
Mutations bypass the server-side SQL compiler and parser entirely. They write values directly to storage partition splits, which makes them faster and more lightweight. However, you cannot read mutations in-flight inside the same transaction before a commit.
DML statements (INSERT, UPDATE, DELETE) run via the SQL compilation engine and can participate interactively inside a transaction, allowing you to read database rows that were modified earlier in the transaction block.

In this step, we use the highly optimized Mutation API to write data into the interleaved TradingAccounts and Trades tables.

Append this helper function to your code and invoke it inside main():

def write_mutations(database):
    # Mutations are applied atomically in a batch

func writeMutations(ctx context.Context, client *spanner.Client) error {
	accountCols := []string{"AccountId", "AccountName", "FirmName"}
	tradeCols := []string{"AccountId", "TradeId", "ProductSymbol", "NotionalValue"}

	mutations := []*spanner.Mutation{
		// Insert or Update parent records
		spanner.InsertOrUpdate("TradingAccounts", accountCols, []interface{}{1, "CME_MAIN_ACC", "CME Group Client Services"}),
		spanner.InsertOrUpdate("TradingAccounts", accountCols, []interface{}{2, "CME_RISK_ACC", "Clearing Risk Management"}),
		spanner.InsertOrUpdate("TradingAccounts", accountCols, []interface{}{3, "CME_TEST_ACC", "API SandBox Firm"}),
		
		// Insert or Update interleaved child records
		spanner.InsertOrUpdate("Trades", tradeCols, []interface{}{1, 1, "ESU6 (E-mini S&P 500 Futures)", 5000000}),
		spanner.InsertOrUpdate("Trades", tradeCols, []interface{}{1, 2, "CLU6 (Crude Oil Futures)", 2500000}),
		spanner.InsertOrUpdate("Trades", tradeCols, []interface{}{2, 1, "GCU6 (Gold Futures)", 3500000}),
	}

	// Apply mutations batch atomically
	_, err := client.Apply(ctx, mutations)
	if err == nil {
		fmt.Println("Inserted rows using Mutations successfully!")
	}
	return err
}

Step 4: Executing Queries using SQL

Cloud Spanner supports standard ANSI relational SQL queries. When performing queries, you should use parameterized values instead of string interpolation to prevent SQL injection vulnerabilities and maximize cache reuse for query execution plans.

Define a query helper function in main.py:

def query_data(database):
    # Queries are executed in read-only snapshots to ensure strong consistency
    with database.snapshot() as snapshot:
        query = "SELECT AccountId, AccountName, FirmName FROM TradingAccounts ORDER BY AccountId"
        results = snapshot.execute_sql(query)
        
        print("\nQuery Results (TradingAccounts):")
        for row in results:
            print(f"ID: {row[0]}, Name: {row[1]} (Firm: {row[2]})")

import "google.golang.org/api/iterator" // Add to imports
 
func queryData(ctx context.Context, client *spanner.Client) error {
	stmt := spanner.Statement{SQL: "SELECT AccountId, AccountName, FirmName FROM TradingAccounts ORDER BY AccountId"}
	
	// Open read-only transaction snapshot
	iter := client.Single().Query(ctx, stmt)
	defer iter.Stop()

	fmt.Println("\nQuery Results (TradingAccounts):")
	for {
		row, err := iter.Next()
		if err == iterator.Done {
			break
		}
		if err != nil {
			return err
		}
		
		var id int64
		var accountName, firmName string
		if err := row.Columns(&id, &accountName, &firmName); err != nil {
			return err
		}
		fmt.Printf("ID: %d, Name: %s (Firm: %s)\n", id, accountName, firmName)
	}
	return nil
}

Step 5: Managing Read-Write Transactions

Cloud Spanner supports ACID transactions. To execute modifications securely, you must use a **Read-Write Transaction**. When writing client code, you specify a transaction callback function. The SDK client libraries handle the necessary pessimistic locks and automatically retry the entire block if a conflict forces a retry.

Deep Dive: Auto-Retry Mechanism
Because Spanner runs transactions with pessimistic locks, lock conflicts on resource keys can trigger database abort signals. The transaction runner function is designed to catch these abort signals, release locks, reset resources, and transparently replay the callback function. Rule of thumb: never include non-idempotent code (like sending an email or charging a payment API) directly inside the transaction callback, as it might run multiple times.

In this step, we implement a transaction block that reads the current NotionalValue value for a specific trade, adds value size adjustment, and executes a database UPDATE using parameterized DML statements.

Implement the transaction callback function in main.py:

main.py (Additions)
Copied! 
def run_transaction(database):
    # Define callback task function
    def update_trade_value(transaction):
        # 1. Fetch current NotionalValue inside transaction scope
        results = transaction.execute_sql(
            "SELECT NotionalValue FROM Trades WHERE AccountId = 1 AND TradeId = 1"
        )
        row = list(results)[0]
        current_val = row[0]
        
        # Compute adjustment
        new_val = current_val + 500000
        
        # 2. Update database record using SQL DML
        transaction.execute_update(
            "UPDATE Trades SET NotionalValue = @notional WHERE AccountId = 1 AND TradeId = 1",
            params={"notional": new_val},
            param_types={"notional": spanner.param_types.INT64}
        )
        print(f"Transaction complete! Updated NotionalValue from {current_val} to {new_val}")

    # Run callback through helper which wraps transaction block
    database.run_in_transaction(update_trade_value)

async function runTransaction(database) {
  await database.runTransactionAsync(async (transaction) => {
    // 1. Fetch current NotionalValue inside transaction scope
    const [rows] = await transaction.run({
      sql: 'SELECT NotionalValue FROM Trades WHERE AccountId = 1 AND TradeId = 1'
    });

    const currentVal = parseInt(rows[0][0].value);
    const newVal = currentVal + 500000;

    // 2. Perform value update using database DML statement
    await transaction.runUpdate({
      sql: 'UPDATE Trades SET NotionalValue = @notional WHERE AccountId = 1 AND TradeId = 1',
      params: {
        notional: newVal
      },
      types: {
        notional: 'int64'
      }
    });

    console.log(`Transaction complete! Updated NotionalValue from ${currentVal} to ${newVal}`);
    await transaction.commit();
  });
}

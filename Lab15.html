Step 2: Recommendation 1 – Aborted Transaction Retry (`@Retryable`)

Automatically retry read-write transactions that are aborted by Spanner's lock manager under lock contention.

📖 Official Google Spanner Documentation:
"Read-write transactions in Cloud Spanner use pessimistic locks... Cloud Spanner may abort a read-write transaction at any time due to lock contention. When a transaction is aborted, the application MUST retry the transaction from the beginning."

Code Implementation (SpannerBankApplication.java & BankingService.java)

Java Code Implementation
// 1. SpannerBankApplication.java:
@SpringBootApplication
@EnableRetry // Enables Spring Retry framework
public class SpannerBankApplication { ... }

// 2. BankingService.java:
@Retryable(
    retryFor = { Exception.class },
    maxAttempts = 5,
    backoff = @Backoff(delay = 100, multiplier = 2.0)
)
@Transactional
public PaymentTransaction transferMoney(String sourceAccountId, String targetAccountId, BigDecimal amount) { ... }

Step 3: Recommendation 2 – `SELECT FOR UPDATE` Lock Escalation Prevention

Acquire exclusive write locks during initial read phase to prevent lock escalation conflicts.

📖 Official Google Spanner Documentation:
"Reading a row in a read-write transaction acquires a shared read lock. Updating the row later requires escalating to an exclusive write lock. Using SELECT FOR UPDATE acquires an exclusive lock immediately on read, avoiding lock escalation aborts."

Code Implementation (AccountRepository.java & BankingService.java)

Java Code Implementation
// 1. AccountRepository.java:
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT a FROM Account a WHERE a.accountId = :accountId")
Optional<Account> findByAccountIdForUpdate(@Param("accountId") String accountId);

// 2. BankingService.java:
Account sourceAccount = accountRepository.findByAccountIdForUpdate(sourceAccountId).orElseThrow();
Account targetAccount = accountRepository.findByAccountIdForUpdate(targetAccountId).orElseThrow();
  • Inspect SELECT FOR UPDATE Verify LockModeType.PESSIMISTIC_WRITE on repository findByAccountIdForUpdate method.

Step 4: Recommendation 3 – Physical Table Interleaving (`INTERLEAVE IN PARENT`)

Co-locate child account rows physically inside the parent customer split node.

📖 Official Google Spanner Documentation:
"Interleaving is a powerful feature in Cloud Spanner for parent-child relationships... Interleaving physically stores child table rows with the parent row in the same split node. This eliminates network RPC latency when joining parent and child tables."

Step 5: Recommendation 4 – Secondary Covering Indexes (`INCLUDE` Clause)

Include non-key payload columns in secondary indexes to eliminate 2-step table backjoins using PostgreSQL dialect INCLUDE clause.

📖 Official Google Spanner Documentation:
"Using the INCLUDE clause in PostgreSQL dialect (STORING in Google Standard SQL) includes additional payload columns in the index. This turns the index into a covering index, allowing Spanner to satisfy queries directly from the index split without an expensive backjoin to the base table split."

DDL Implementation (ddl/schema.sql)

PostgreSQL Dialect DDL
-- Covering Index for Accounts by Account Type (PostgreSQL dialect uses INCLUDE):
CREATE INDEX IF NOT EXISTS idx_accounts_type 
ON accounts(account_type) 
INCLUDE (currency, balance, updated_at);

-- Covering Index for Transactions by Source Account:
CREATE INDEX IF NOT EXISTS idx_payments_source 
ON payment_transactions(source_account_id) 
INCLUDE (target_account_id, amount, status, created_at);

Step 8: Recommendation 7 – Connection Pool & Sizing Rules for 100 PUs

Understand the 6 official Cloud Spanner connection management rules and why specific HikariCP pool values were chosen for a 100 Processing Units (0.1 Node) instance.

📖 Official Google Spanner Documentation:
"Creating a new Spanner session involves TLS handshakes, gRPC channel initialization, and server-side session allocation. Applications should maintain a long-lived connection pool and size maximum pool limits according to Spanner node/PU compute capacity."

Google Cloud Spanner 6 Connection Pooling Rules

1. Reuse a Single Long-Lived Connection Pool per JVM
Recommendation: Initialize one single DataSource / HikariCP connection pool per application process and reuse it for the entire application lifecycle. Never open/close connections or create new pools per HTTP request.
Why: Creating a new connection requires TLS handshakes, gRPC channel setup, and Spanner session creation (spanner.sessions.create), adding 100ms–300ms of latency per request.
2. Match Pool Sizing to Spanner Capacity (100 PUs = 20-30 Connections)
Recommendation: Set maximum-pool-size based on compute capacity. For 100 PUs (0.1 Node), set maximum-pool-size: 25. For 1 Full Node (1,000 PUs), set maximum-pool-size: 50-100.
Why: Oversizing connection pools (e.g. setting pool size to 500 on a small 100 PU instance) causes session lock contention inside Spanner's internal session pool manager.
3. Session Pre-Warming (minimum-idle: 10)
Recommendation: Configure minimum-idle: 10 in HikariCP.
Why: Pre-warms gRPC network channels and Spanner sessions on application startup, eliminating cold-start latency spikes when initial user traffic hits the microservice.
4. Max Lifetime Below Spanner Session Expiry (max-lifetime: 30 Mins)
Recommendation: Set HikariCP max-lifetime: 1800000 (30 minutes) and idle-timeout: 600000 (10 minutes).
Why: Cloud Spanner sessions automatically expire server-side after 60 minutes of inactivity. Setting max-lifetime to 30 minutes ensures idle connections are refreshed by HikariCP before Spanner forcibly terminates them.
5. Connection Keep-Alive & Health Check (connection-test-query: SELECT 1)
Recommendation: Configure connection-test-query: SELECT 1.
Why: Detects and evicts dead TCP connections caused by cloud network firewalls or load balancer idle timeouts before handing connections to application worker threads.
6. gRPC HTTP/2 Channel Multiplexing
Recommendation: PGAdapter and Spanner Client libraries automatically multiplex up to 100 concurrent database sessions over a small number of underlying HTTP/2 gRPC channels (default: 4 channels).
Why: Prevents TCP socket exhaustion and maximizes network throughput over high-latency WAN links.

application.yml HikariCP Implementation

application.yml (HikariCP Settings for 100 PUs)

    hikari:
      maximum-pool-size: 25   # Rule 2: Size tuned for 100 PUs (20-30 max) to prevent session lock contention

      idle-timeout: 600000    # Rule 4: 10 mins idle eviction limit
      max-lifetime: 1800000   # Rule 4: 30 mins max lifetime (below Spanner's 60-min session expiry)
      connection-timeout: 30000 # Fail-fast 30s wait limit

Step 10: Recommendation 9 – Spanner Mutations vs SQL DML Batching

Bypass SQL query planning for high-throughput write ingestion pipelines using PGAdapter binary Mutation batching without sacrificing SQL query optimization.

📖 Official Google Spanner Documentation:
"Mutations write raw column tuples directly to storage, bypassing SQL parsing and plan compilation. For high-throughput bulk write operations, Mutations offer 2x to 3x higher throughput and lower CPU consumption compared to SQL DML."
Architectural Deep Dive: Do We Lose SQL Engine Optimization?
NO! You get the best of both worlds. Setting preferQueryMode=extended tells PGAdapter to use PostgreSQL Extended Query Protocol (Parse → Bind → Execute):
  • Complex SELECT, UPDATE & DELETE Queries: Execute 100% via Spanner's SQL Query Engine & Cost-Based Optimizer (spanner.optimizer_version=LATEST) with cached query plans for maximum query performance.
  • Simple JDBC Batch Inserts (saveAll()): Simple record writes without WHERE clauses or joins are converted into Spanner Mutations, writing raw key-value tuples directly to storage for 2x–3x higher ingestion throughput.

Execution Engine & Optimizer Comparison Matrix

Spanner Execution Matrix
+------------------------------------+------------------------------------+----------------------------+-------------------------+
| Statement Type                     | Execution Engine in Spanner        | Cost-Based Optimizer Used? | Cached Query Plans?     |
+------------------------------------+------------------------------------+----------------------------+-------------------------+
| SELECT Queries                     | Spanner SQL Query Engine           | YES (optimizer_version)    | YES (Extended Protocol) |
| Complex UPDATE / DELETE            | Spanner SQL Query Engine           | YES (optimizer_version)    | YES (Extended Protocol) |
| Partitioned DML                    | Distributed Spanner Query Engine   | YES                        | YES                     |
| Simple saveAll() Batch Inserts     | Spanner Storage Mutation API       | Direct Key-Value Storage   | Direct Key-Value Storage|

Step 11: Multi-Table High-Performance JDBC Batching

Batch insert Customers AND Interleaved Accounts in single multi-table network RPC calls.

BankingService.java
@Transactional
public List<Customer> seedBulkCustomers(int count) {
    List<Customer> customerList = new ArrayList<>();
    List<Account> accountList = new ArrayList<>();

    for (int i = 1; i <= count; i++) {
        String customerId = UUID.randomUUID().toString();
        Customer c = Customer.builder().customerId(customerId).fullName("User " + i).email("u" + i + "@bank.com").status("ACTIVE").build();
        customerList.add(c);

        Account a = Account.builder().customerId(customerId).accountId(UUID.randomUUID().toString()).accountType("CHECKING").currency("USD").balance(BigDecimal.valueOf(1000.00)).build();
        accountList.add(a);
    }

    List<Customer> saved = customerRepository.saveAll(customerList);
    accountRepository.saveAll(accountList);
    return saved;
}

Step 12: Non-Locking Snapshot Reads

Serve read queries from local read-replicas without acquiring locks using @Transactional(readOnly = true).

📖 Official Google Spanner Documentation:
"Read-only transactions (Snapshot Reads) do not acquire locks and do not block read-write transactions... Because read-only transactions execute without lock overhead, they offer higher read throughput and eliminate lock contention between concurrent readers and writers."

BankingService.java Code Implementation

BankingService.java
// Non-Locking Snapshot Read: Eliminates read/write lock contention
@Transactional(readOnly = true)
public List<Customer> getAllCustomers() {
    return customerRepository.findAll();
}

@Transactional(readOnly = true)
public List<Account> getAccountsByCustomer(String customerId) {
    return accountRepository.findByCustomerId(customerId);
}

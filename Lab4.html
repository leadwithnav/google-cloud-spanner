Refactoring a Legacy Schema for Spanner

Welcome! In this hands-on lab, you will learn how to refactor a traditional relational database schema into an optimized schema built for Google Cloud Spanner's distributed architecture.

The Distributed Schema Challenge

In a traditional relational database (like PostgreSQL or MySQL), you design schemas with normalized tables linked by foreign keys. In a single-instance database, joins across these tables are computationally cheap because all data resides on the same machine.

However, Spanner divides tables into chunks called splits and distributes them across multiple server nodes. In a naive schema, joining two tables (e.g. Customers and Orders) requires joining splits across different physical servers, which causes heavy network latency and slows down operations.

The Spanner Optimization Strategy
To achieve maximum performance at scale in Spanner, you must design your schema to co-locate related data on the same physical split. We achieve this using Table Interleaving and covered secondary indexes (using the STORING clause).

Lab Objectives

In this lab, you will refactor a legacy database structure into an optimized Spanner design by doing the following:

  • Analyze a legacy normalized schema containing Customers, Orders, and OrderItems.
  • Refactor the tables into a hierarchical, **interleaved structure**.
  • Design secondary indexes to handle query filtering.
  • Apply **Covered Indexes** (using the `STORING` option) to eliminate back-join latency.
  • Deploy the schema to your local emulator database and execute validation queries.
  • Understand the goals of the lab Review why traditional normalized database schemas are suboptimal in distributed environments.

Step 1: Analyzing the Legacy Schema

Let's study a typical e-commerce relational schema designed for a traditional single-instance database. Examine the SQL declarations below:

SQL (Legacy Normalized Schema)
Copied! 
-- Parent Customers table
CREATE TABLE Customers (
  CustomerId INT PRIMARY KEY,
  Name VARCHAR(255),
  Email VARCHAR(255),
  Country VARCHAR(100)
);

-- Orders table linked by foreign key
CREATE TABLE Orders (
  OrderId INT PRIMARY KEY,
  CustomerId INT,
  OrderDate DATE,
  TotalAmount DECIMAL(10, 2),
  FOREIGN KEY (CustomerId) REFERENCES Customers(CustomerId)
);

-- OrderItems table linked by foreign key to Orders
CREATE TABLE OrderItems (
  OrderItemId INT PRIMARY KEY,
  OrderId INT,
  ProductId INT,
  Quantity INT,
  Price DECIMAL(10, 2),
  FOREIGN KEY (OrderId) REFERENCES Orders(OrderId)
);

Why this underperforms on Spanner:

  1. Data Scattering: Rows for Customer `101`, their Orders, and their Order Items will likely be stored on different physical servers.
  2. Distributed Joins: Running a query to calculate order totals with customer information requires gathering rows from separate splits over the network, causing latency.
  3. Index Inefficiencies: Querying orders by `OrderDate` requires scanning the entire `Orders` table, or incurring distributed secondary index updates.
  • Identify schema design pain points Analyze how foreign key relationships partition data in a distributed network.

Step 2: Refactoring with Table Interleaving

To optimize the legacy schema, we will define a parent-child relationship in Spanner's storage configuration using the INTERLEAVE IN PARENT clause.

Designing Interleaved Primary Keys

For Spanner to co-locate child records, the Primary Key of the child table must start with the Primary Key of the parent table. Observe the structural mapping:

  • Customers (Root Parent): PK is (CustomerId)
  • Orders (Child of Customers): PK is (CustomerId, OrderId)
  • OrderItems (Grandchild / Child of Orders): PK is (CustomerId, OrderId, OrderItemId)
Primary Key Implication
Notice that OrderId alone is no longer the primary key of the Orders table. The compound key (CustomerId, OrderId) is required. If your application queries orders by OrderId without knowing the CustomerId, you will need a secondary index. We will design this index in the next step.

The Refactored DDL

Here is the refactored schema. Notice how the primary keys are configured and how the INTERLEAVE and ON DELETE CASCADE options are applied:

SQL (Interleaved Schema DDL)
Copied! 
CREATE TABLE Customers (
  CustomerId INT64 NOT NULL,
  Name STRING(255),
  Email STRING(255),
  Country STRING(100),
) PRIMARY KEY (CustomerId);

CREATE TABLE Orders (
  CustomerId INT64 NOT NULL,
  OrderId INT64 NOT NULL,
  OrderDate DATE,
  TotalAmount NUMERIC,
) PRIMARY KEY (CustomerId, OrderId),
INTERLEAVE IN PARENT Customers ON DELETE CASCADE;

CREATE TABLE OrderItems (
  CustomerId INT64 NOT NULL,
  OrderId INT64 NOT NULL,
  OrderItemId INT64 NOT NULL,
  ProductId INT64,
  Quantity INT64,
  Price NUMERIC,
) PRIMARY KEY (CustomerId, OrderId, OrderItemId),
INTERLEAVE IN PARENT Orders ON DELETE CASCADE;
  • Design compound primary keys Verify that child tables inherit the primary keys of parent tables.
  • Understand cascade behaviors Verify how 'ON DELETE CASCADE' affects child records during deletion.

Step 3: Optimizing Indexes & covered columns

In Spanner, secondary indexes are structured as tables behind the scenes. When you query using an index, Spanner searches the index table for the key, fetches the corresponding primary key, and then queries the primary table to get other requested columns. This second fetch is called a **back-join**.

Deep Dive: Covered Indexes with STORING
To prevent back-joins, you can duplicate columns from the main table directly into the index using the STORING clause. When a query requests stored columns, Spanner returns them directly from the index, eliminating the back-join entirely and improving read performance.

Designing the Index

We want to support two queries efficiently:

  1. Find all orders made on a specific date: SELECT CustomerId, OrderId FROM Orders WHERE OrderDate = '2026-07-17'
  2. Find total sales on a date without back-joining the main table: SELECT CustomerId, TotalAmount FROM Orders WHERE OrderDate = '2026-07-17'

Let's write DDL statements to create these indexes:

SQL (Index DDL)
Copied! 
-- Standard index on OrderDate (creates back-joins for TotalAmount query)
CREATE INDEX OrdersByDate ON Orders(OrderDate);

-- Optimized covered index storing TotalAmount
CREATE INDEX OrdersByDateCovered ON Orders(OrderDate) STORING (TotalAmount);
  • Analyze back-join latency Verify how back-joins impact query speed in distributed clusters.
  • Incorporate STORING clauses Determine which fields should be stored in the index based on common query patterns.

Step 4: Deploying the Schema

Now, let's deploy the refactored schema and indexes to our local database emulator using the gcloud spanner databases ddl update tool.

Execute the DDL Update

Run the following command in your terminal to create the tables and secondary indexes:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database \
  --instance=test-instance \
  --ddl='CREATE TABLE Customers (CustomerId INT64 NOT NULL, Name STRING(255), Email STRING(255), Country STRING(100)) PRIMARY KEY (CustomerId); CREATE TABLE Orders (CustomerId INT64 NOT NULL, OrderId INT64 NOT NULL, OrderDate DATE, TotalAmount NUMERIC) PRIMARY KEY (CustomerId, OrderId), INTERLEAVE IN PARENT Customers ON DELETE CASCADE; CREATE TABLE OrderItems (CustomerId INT64 NOT NULL, OrderId INT64 NOT NULL, OrderItemId INT64 NOT NULL, ProductId INT64, Quantity INT64, Price NUMERIC) PRIMARY KEY (CustomerId, OrderId, OrderItemId), INTERLEAVE IN PARENT Orders ON DELETE CASCADE; CREATE INDEX OrdersByDate ON Orders(OrderDate); CREATE INDEX OrdersByDateCovered ON Orders(OrderDate) STORING (TotalAmount);'

Verify Schema Registration

Verify that all tables and indexes are successfully registered in the database metadata:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl describe test-database \
  --instance=test-instance

Step 5: Verifying performance and local joins

Let's insert sample records and examine how queries leverage table interleaving and stored indexes.

1. Populate Sample Records

Populate the database tables using DML statements:

Terminal (Bash / PowerShell)
Copied! 
# Insert Customer
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="INSERT INTO Customers (CustomerId, Name, Email, Country) VALUES (101, 'Alice Smith', 'alice@email.com', 'Germany')"

# Insert Interleaved Order
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="INSERT INTO Orders (CustomerId, OrderId, OrderDate, TotalAmount) VALUES (101, 5001, '2026-07-17', 150.00)"

# Insert Interleaved Item
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="INSERT INTO OrderItems (CustomerId, OrderId, OrderItemId, ProductId, Quantity, Price) VALUES (101, 5001, 1, 9901, 2, 75.00)"

2. Run Co-located Join Query

Run a query that joins Customers, Orders, and OrderItems. Because the schema is interleaved, Spanner can evaluate this join locally on a single server split without network hops:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="SELECT c.Name, o.OrderId, o.OrderDate, i.ProductId, i.Quantity, i.Price FROM Customers c JOIN Orders o ON c.CustomerId = o.CustomerId JOIN OrderItems i ON o.CustomerId = i.CustomerId AND o.OrderId = i.OrderId WHERE c.CustomerId = 101"

3. Force the Covered Index

Run a query targeting `OrderDate`. Use a query hint to force Spanner to run the query using our index. The database will return the `TotalAmount` directly from the index table without performing a back-join:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="SELECT CustomerId, TotalAmount FROM Orders@{FORCE_INDEX=OrdersByDateCovered} WHERE OrderDate = '2026-07-17'"

  • Test index performance Verify that queries using stored index values return correctly.

Profiling Query Executions & Optimizing with Covered Indexes

Welcome! In this hands-on lab, you will learn how to analyze slow-running queries using Spanner's Query Execution Plan, identify performance bottlenecks like full table scans and back-joins, and optimize latency using storing secondary indexes.

What is an Execution Plan?

Before running an SQL query, Google Cloud Spanner compiles it into an **Execution Plan**. The plan is a tree structure composed of database operators (like *Distributed Union*, *Index Scan*, or *Hash Join*). Examining this plan helps developers understand exactly how Spanner partitions and queries tables across servers.

Rather than guessing why a query runs slowly, we can profile it using the command-line flag --query-mode=PROFILE, which prints actual diagnostic statistics (CPU execution time, data rows scanned, and memory footprint) alongside the plan nodes.

The Tuning Strategy
We will start with a query that triggers a slow **Full Table Scan**. We will then add a standard index and diagnose a secondary bottleneck: the **Back-Join** (or key lookup). Finally, we'll eliminate the back-join using a covered **Storing Index**, achieving sub-millisecond execution times.

Step 1: Setting Up the Test Scenario

Let's define a new interleaved table named Transactions to record card purchases. Deploy this schema definition onto your active emulator instance.

1. Deploy the DDL Schema

Execute the DDL update in your terminal to append the Transactions table to your database. Note that it is interleaved under Customers (reusing our Customer hierarchy from Lab 3):

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database \
  --instance=test-instance \
  --ddl="CREATE TABLE Transactions (
    CustomerId INT64 NOT NULL,
    TransactionId INT64 NOT NULL,
    TransactionDate DATE,
    Amount NUMERIC,
    Merchant STRING(255),
    Status STRING(50)
  ) PRIMARY KEY (CustomerId, TransactionId),
  INTERLEAVE IN PARENT Customers ON DELETE CASCADE;"

2. Populate Transaction Records

Let's add test transactions to query against:

Step 2: Profiling the Naive Query

Imagine your user dashboard queries the transactions table to list all successful purchases at `'Cloud Cafe'` on a specific date:

SELECT CustomerId, Amount, Status FROM Transactions WHERE Merchant = 'Cloud Cafe' AND TransactionDate = '2026-07-17'

1. Run the Profile Command

Profile the execution of this query by adding the --query-mode=PROFILE option. This tells Spanner to capture query execution statistics and output the query execution plan:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --query-mode=PROFILE \
  --sql="SELECT CustomerId, Amount, Status FROM Transactions WHERE Merchant = 'Cloud Cafe' AND TransactionDate = '2026-07-17'"

2. Analyze the Output Plan

Review the printed plan output in your terminal. You will notice plan operators similar to this:

Execution Plan Concept 
Distributed Union
└── Local Distributed Union
    └── Filter (Condition: Merchant = 'Cloud Cafe' AND TransactionDate = '2026-07-17')
        └── Table Scan (Table: Transactions)
Bottleneck: Table Scan Operator
Because there is no index on Merchant or TransactionDate, Spanner must read every single row in the Transactions table to check if it matches the filter. In a database with millions of rows, this full table scan causes heavy CPU load and high latency.
  • Run the query with profile enabled Verify that the gcloud CLI outputs the execution plan and run statistics.
  • Identify the Table Scan operator Locate the Table Scan step in the output plan.

Step 3: Diagnosing Back-Join Bottlenecks

Let's add a standard secondary index to speed up the query filtering. We will then analyze the new execution plan to identify any remaining bottlenecks.

1. Create a Standard Secondary Index

Deploy a standard index on Merchant and TransactionDate:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database \
  --instance=test-instance \
  --ddl="CREATE INDEX TransactionsByMerchantDate ON Transactions(Merchant, TransactionDate);"

2. Profile the Query Again

Run the profile command a second time to see how the plan has changed:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --query-mode=PROFILE \
  --sql="SELECT CustomerId, Amount, Status FROM Transactions WHERE Merchant = 'Cloud Cafe' AND TransactionDate = '2026-07-17'"

3. Examine the Back-Join (Key Lookup) Plan

The output plan tree should look similar to this:

Execution Plan Concept 
Distributed Cross Apply
├── Index Scan (Index: TransactionsByMerchantDate, Condition: Merchant = 'Cloud Cafe')
└── Key Lookup (Table: Transactions, Fetch: Amount, Status)
Bottleneck: Key Lookup (Back-Join)
Spanner uses the index to quickly find rows matching Merchant = 'Cloud Cafe'. However, because the index table does not contain the Amount and Status columns, Spanner must perform a back-join to fetch those values from the primary table. At scale, this back-join cross-network lookup becomes a major performance bottleneck.
  • Deploy the standard index Verify that the standard index on Merchant and TransactionDate is created.
  • Identify the Key Lookup step Locate the Distributed Cross Apply and Key Lookup operators in the query profile output.

Step 4: Deploying a Storing Covered Index

To eliminate the back-join, we will replace the standard index with a **Storing Covered Index**. This copies the query's returned columns directly into the secondary index table.

1. Drop the Standard Index

Remove the suboptimal standard index first:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database \
  --instance=test-instance \
  --ddl="DROP INDEX TransactionsByMerchantDate;"

2. Create the Storing Covered Index

Create a new index on the filter columns (Merchant, TransactionDate) and store the returned columns Amount and Status using the STORING clause:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases ddl update test-database \
  --instance=test-instance \
  --ddl="CREATE INDEX TransactionsByMerchantDateCovered 
         ON Transactions(Merchant, TransactionDate) 
         STORING (Amount, Status);"
  • Drop the standard index Remove the TransactionsByMerchantDate index.
  • Deploy the Storing Covered Index Deploy the new index using the STORING clause to include Amount and Status values.

Step 5: Verifying the Optimization

Now, let's profile the query again to verify that the back-join is gone and the execution plan is optimized.

1. Profile the Query a Third Time

Run the query profile command one last time:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --query-mode=PROFILE \
  --sql="SELECT CustomerId, Amount, Status FROM Transactions WHERE Merchant = 'Cloud Cafe' AND TransactionDate = '2026-07-17'"

2. Analyze the Final Covered Plan

The output plan tree should look similar to this:

Execution Plan Concept 
Distributed Union
└── Local Distributed Union
    └── Index Scan (Index: TransactionsByMerchantDateCovered, Condition: Merchant = 'Cloud Cafe')
Success: No Key Lookup Operator!
Observe that the **Key Lookup** step is gone. Spanner retrieved all requested fields directly from the index, achieving sub-millisecond query latency.
  • Profile the optimized query Verify that the execution plan uses Index Scan and has no Key Lookup step.
  • Compare performance stats Review the reduction in CPU execution time and rows scanned.

Step 6: Deep Dive: Spanner Key Visualizer

Learn how to read Spanner's graphical Key Visualizer heatmaps to diagnose distributed bottlenecks like key hotspots and lock contention.

What is Key Visualizer?

Google Cloud Spanner automatically shards tables and indexes into physical splits based on load and key ranges. If an application concentrates high throughput on a single range of keys, that split's hosting server becomes overloaded. Key Visualizer is a console diagnostic tool that renders interactive heatmaps of database traffic over time, helping developers locate these bottlenecks.

Heatmap Layout
X-Axis (Time): Shows hours or days of execution.
Y-Axis (Key Space): Shows table, index, and row primary key ranges sorted lexicographically.
Brightness/Color: Represents traffic intensity (darker is quiet, bright red/white is heavy traffic).
Metrics Visualized
Ops/second: Shows read/write request density.
CPU utilization: Highlights nodes running hot.
Lock conflicts: Pinpoints lock wait times and write contention bottlenecks.

1. Identifying Row & Key Hotspots

A **hotspot** occurs when reads or writes concentrate in a tight key range. Because Spanner is distributed, this overloads a single node while others remain idle.

Visual Signature

Hotspots appear as bright horizontal bands or diagonal streaks on heatmaps:

  • Bright Horizontal Line: Indicates constant high CPU or writes to a single row (e.g., a global database counter or single record updated continuously).
  • Diagonal Streak (Staircase): Indicates monotonically increasing primary keys (e.g., sequential integer IDs or timestamps). As new keys are inserted, the active split location shifts linearly, leaving a bright trail.
Hotspotting Mitigation
Avoid sequential primary keys. Instead:
  1. Use universally unique identifiers (e.g. UUIDv4) to distribute writes.
  2. Hash monotonically increasing keys before insertion.
  3. Shard keys by prepending a calculated shard ID (e.g. hash(id) % SHARD_COUNT).

2. Identifying Lock Contention

When multiple parallel transactions try to write to the same rows, Spanner serializes them using locks. If transactions take too long or compete heavily, you will experience **Lock Contention** (resulting in high abort/retry counts and latency spikes).

Visual Signature

In Key Visualizer, switch the metric filter to Lock conflicts. Contention will highlight as bright horizontal stripes covering the specific hot keys range during high transaction concurrency.

Lock Contention Mitigation
  • Use Read-Only Transactions: Read-only transactions do not take locks and run lock-free across replicas.
  • Optimize Write Scope: Keep read-write transactions small and avoid long-running calculations within transaction callbacks.
  • Batch Updates: Batch mutations where possible to minimize lock holds.
  • Analyze Key Visualizer layouts Understand how key spaces and timestamps map to heatmap axes.
  • Learn hotspot & lock contention patterns Review visual patterns of sequential primary keys and transaction serialization contention.

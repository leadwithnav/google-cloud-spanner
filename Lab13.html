Data Placements in Cloud Spanner

Welcome! In this hands-on lab, you will learn how to configure **Data Placements** and **Geo-Partitioning** in Cloud Spanner to satisfy strict data residency requirements and optimize read/write latencies globally.

Why Data Placements?

In globally distributed systems, storing all database records in a single region limits write latency for remote users and can violate local data sovereignty regulations (such as GDPR in Europe). Spanner's **Data Placements** (available in Spanner Enterprise Plus) solve this by partitioning data at the row level across regions:

  • Data Sovereignty: Pin European users' records to Europe-based replication servers and American users' records to US-based servers.
  • Localized Latency: Ensure clients read and write to nodes physically nearest to them, achieving near single-digit millisecond performance.
  • Unified Schema: Maintain a single global table schema that is automatically partitioned under the hood.
Key Concept: Placement Key

A Placement Key is a column attribute declared during table creation that tells Spanner which replica partition should physically host the row. To route queries efficiently, the placement key must be part of the table's Primary Key.

Creating Instance Partitions

Before placing database rows, you must allocate compute capacity in the target regions. You achieve this by creating **Instance Partitions**.

Instance Partition Architecture

An instance partition represents a distinct slice of computing power (nodes or processing units) tied to a specific geographic region inside your multi-region instance configuration.

1. Provision Multi-Region Instance

Run this command to create a multi-region Spanner instance allocated with 200 Processing Units (PUs):

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner instances create test-instance \
  --config=nam-eur-asia1 \
  --description="Global Multi-region Instance" \
  --nodes=2

2. Divide Nodes into 2 Partitions

Now, divide those 2 nodes equally by creating two instance partitions of 1 node each (assigning one partition to the Americas and one to Europe):

Important: No Granular Partitions

Cloud Spanner does not support granular instances (capacities in processing units less than 1000 PUs / 1 node) or processing units for the geo-partitioning / instance partition features. Partitions must always be configured with at least 1 full node (e.g. --nodes=1).

Terminal (Bash / PowerShell)
Copied! 
# 1. Create Americas partition with 1 node
gcloud spanner instance-partitions create us-partition \
  --instance=test-instance \
  --config=regional-us-east1 \
  --nodes=1 \
  --description="Americas regional partition"

# 2. Create Europe partition with 1 node
gcloud spanner instance-partitions create eu-partition \
  --instance=test-instance \
  --config=regional-europe-west3 \
  --nodes=1 \
  --description="Europe regional partition"

Creating Database Placements

Once regional compute resource boundaries exist, register them as **Placements** within the database schema.

DDL: CREATE PLACEMENT

Placements map database schema partitions to physical instance partition containers. Run the following DDL statements inside your database schema:

DDL
Copied! 
-- Register US placement
CREATE PLACEMENT placement_us 
OPTIONS (
  instance_partition = 'us-partition'
);

-- Register Europe placement
CREATE PLACEMENT placement_eu 
OPTIONS (
  instance_partition = 'eu-partition'
);
  • Register database placements DDL Link logical database placement IDs to regional instance partition configurations.

Declaring Table Placement Keys

Now, deploy a schema that distributes customer rows based on the placement keys.

1. Deploy Geo-Partitioned Table

Define a table containing a column decorated with the PLACEMENT KEY constraint. Run the DDL script to create the Customers table:

DDL
Copied! 
CREATE TABLE Customers (
  CustomerId INT64 NOT NULL,
  Name STRING(255),
  Email STRING(255),
  Region STRING(50) NOT NULL PLACEMENT KEY
) PRIMARY KEY (Region, CustomerId);

2. Insert Regional Records

Write records directly into their respective geographic regions by passing the correct placement IDs in the Region column:

Terminal (Bash / PowerShell)
Copied! 
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="INSERT INTO Customers (CustomerId, Name, Email, Region) VALUES 
  (101, 'Jane Smith', 'jane@email.com', 'placement_us'),
  (102, 'Robert Johnson', 'robert@email.com', 'placement_eu');"
  • Deploy table with PLACEMENT KEY Deploy the schema containing a designated PLACEMENT KEY column constraint.
  • Seed geo-partitioned records Verify that records are routed to their designated regions matching placement configurations.

Observing Placements metadata

You can audit placements and their options by querying Spanner's built-in INFORMATION_SCHEMA.

Query Metadata Tables

Run these SELECT statements to inspect placements registered in your database:

Terminal (Bash / PowerShell)
Copied! 
# 1. Query all database placements
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="SELECT PLACEMENT_NAME, SCHEMA_NAME FROM INFORMATION_SCHEMA.PLACEMENTS;"

# 2. Query configured placement options
gcloud spanner databases execute-sql test-database \
  --instance=test-instance \
  --sql="SELECT PLACEMENT_NAME, OPTION_NAME, OPTION_VALUE 
         FROM INFORMATION_SCHEMA.PLACEMENT_OPTIONS;"

Use Case

Complying with regulatory borders requires isolating localized rows. Adding geo-placements pins row data to regional instances while letting developers query it globally lock-free.

Pro Tip

If you update the value of the placement key column (such as changing Location from 'placement_us' to 'placement_eu'), Spanner physically migrates the row asynchronously to the new region partition!

Configure regional placements and build schemas to isolate user records inside dedicated instance replication environments.

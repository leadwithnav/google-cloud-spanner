Step 1: Spanner Splitting & Leader Architecture

Understand automatic vs manual pre-splitting and multi-region default leader placement.

⚡ 1. Why Pre-Create Splits Before Bulk Operations?
Cloud Spanner automatically divides tables into contiguous primary key ranges called splits. During rapid bulk data loads or updates, automatic auto-splitting takes time to detect load and create new split boundaries. Pre-creating manual split points (using gcloud spanner databases splits add) divides the primary key space upfront, allowing concurrent write traffic to be distributed immediately across multiple servers without hotspotting.
🌐 2. Default Leader Region in Multi-Region Instances
In Spanner multi-region configurations (such as nam6), one read-write region is designated as the Default Leader Region. Read-write leaders for all splits are hosted in this region. Choosing or changing the default leader region places transaction leaders closest to your primary application servers, drastically lowering write transaction commit latencies.

  • Understand Default Leader Placement Learn how default leader regions reduce write commit latency for regional clients.

Step 3: Schema & Primary Key Design

Deploy DDL schema matching your database's dialect (GoogleSQL or PostgreSQL).

⚠️ Dialect Syntax Note
If your database was created with GoogleSQL dialect (default), use STRING(64) and PRIMARY KEY (...) at the end. If created with PostgreSQL dialect, use VARCHAR(64) or TEXT.
Option A: GoogleSQL Dialect DDL (Use if DB is GoogleSQL) 
CREATE TABLE IF NOT EXISTS customer_transactions (
    customer_id STRING(64) NOT NULL,
    transaction_id STRING(64) NOT NULL,
    amount NUMERIC NOT NULL,
    currency STRING(3) NOT NULL,
    status STRING(20) NOT NULL,
    created_at TIMESTAMP NOT NULL OPTIONS (allow_commit_timestamp = true)
) PRIMARY KEY (customer_id, transaction_id);
Option B: PostgreSQL Dialect DDL (Use if DB is PostgreSQL) 
CREATE TABLE IF NOT EXISTS customer_transactions (
    customer_id VARCHAR(64) NOT NULL,
    transaction_id VARCHAR(64) NOT NULL,
    amount NUMERIC NOT NULL,
    currency VARCHAR(3) NOT NULL,
    status VARCHAR(20) NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (customer_id, transaction_id)
);

Step 4: Creating Manual Split Points File

Define split point boundaries in a splits file to pre-partition key space prior to bulk loads.

📝 Composite Primary Key & Quota Rules (splits.txt)
Cloud Spanner table customer_transactions has a 2-column composite primary key: PRIMARY KEY (customer_id, transaction_id).

1. Composite Key Match: Split points MUST specify all primary key parts in order: TABLE customer_transactions ('cust-500000', 'tx-100000').
2. Single Quotes: String values MUST use single quotes ('...').
3. 100 PU Quota Limit: On 100 PU instances, Spanner limits manual split creation to 1 split point per minute (exceeding it triggers 429 RESOURCE_EXHAUSTED).
Correct Format: splits.txt (2 Key Parts for Composite Primary Key) 
TABLE customer_transactions ('cust-500000', 'tx-100000')
  • Prepare Splits File Create splits.txt specifying both composite primary key columns: ('cust-500000', 'tx-100000').

Step 6: Inspecting Table User Split Points

Query SPANNER_SYS.USER_SPLIT_POINTS to verify active manual split boundaries.

🔍 Troubleshooting: Why Might USER_SPLIT_POINTS Return 0 Rows?
  • 1. File Syntax Error: Ensure splits.txt uses customer_transactions 'cust-100000' format (each line must include the table name).
  • 2. Table Creation Order: The table customer_transactions must exist in the database before running gcloud spanner databases splits add.
  • 3. Async Propagation Delay: Spanner takes 10 to 30 seconds after the splits add command completes to update physical storage splits in SPANNER_SYS.USER_SPLIT_POINTS.
  • 4. Case Sensitivity: Query SELECT * FROM SPANNER_SYS.USER_SPLIT_POINTS; without a WHERE clause to list all active splits.
Direct Metadata Query SQL 
-- Inspect active user split points in system catalog
SELECT 
    schema_name, 
    table_name, 
    user_split_point, 
    expire_time 
FROM SPANNER_SYS.USER_SPLIT_POINTS;
gcloud Query Command 
gcloud spanner databases execute-sql test-database1 --instance=test-instance1 \
    --sql="SELECT table_name, user_split_point FROM SPANNER_SYS.USER_SPLIT_POINTS"
gcloud Query Command (PowerShell) 
gcloud spanner databases execute-sql test-database1 --instance=test-instance1 `
    --sql="SELECT table_name, user_split_point FROM SPANNER_SYS.USER_SPLIT_POINTS"
  • Query USER_SPLIT_POINTS Catalog Confirm pre-created split points appear in SPANNER_SYS.USER_SPLIT_POINTS.

Step 8: Default Leader Region Architecture

Understand Paxos leader placement and read-write transaction latency optimization.

🌐 Multi-Region Default Leader Mechanics
In Spanner multi-region configurations (e.g. nam6 spanning us-east1, us-east4, us-central1):
  • Spanner maintains Paxos consensus replicas across multiple geographical regions.
  • Setting default_leader explicitly pins the Paxos read-write leaders for all splits to a designated primary region.
  • Placing the leader region in the same Google Cloud region as your compute workloads (e.g. GKE / Compute Engine in us-east1) achieves minimal round-trip latency for write transactions.
  • Review Leader Region Architecture Understand how default_leader placement minimizes Paxos write commit latencies.
⚠️ Instance Config Requirement for default_leader
default_leader settings are ONLY supported on Multi-Region instance configurations (such as nam6, nam3, asi1, eur3).

If your instance is created under a Single-Region configuration (such as regional-asia-south1 or regional-us-east1), Cloud Spanner throws error does not support default_leader settings because all nodes and leaders reside in that single region.
PostgreSQL / GoogleSQL DDL Syntax (Multi-Region Instances Only) 
-- Set Default Leader Region (Valid ONLY for Multi-Region configs like nam6, nam3)
ALTER DATABASE `test-database1` SET OPTIONS (default_leader = 'us-east1');
  • Inspect DDL Leader Statement Verify ALTER DATABASE SET OPTIONS (default_leader = 'us-east1') syntax.

Step 10: Changing Leader Region via gcloud CLI

Execute gcloud spanner databases ddl update to change default leader region online.

Linux / macOS Commands

    --instance=test-instance1 \
    --ddl="ALTER DATABASE `test-database1` SET OPTIONS (default_leader = 'us-east1')"

    --instance=test-instance1 `

Step 13: Enterprise Multi-Region & Pre-Splitting Best Practices

Review architectural rules for bulk ingestion and leader region optimization.

📋 Production Best Practices Summary
  • Pre-Splitting Strategy: Always pre-create split points using gcloud spanner databases splits add before uploading millions of rows to prevent initial node bottlenecking.
  • Primary Key Hashing: Combine pre-splitting with high-entropy primary keys (e.g. UUIDv4 or SHA256 hashes) to avoid sequential monotonically increasing key bottlenecks.
  • Co-locating Workloads & Leaders: Match your application client region with default_leader to achieve ultra-low latency write commits.

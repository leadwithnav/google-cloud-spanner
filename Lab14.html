Spanner Hotspots and Key Design

In this lab you will explore how a poor primary key design can create write hotspots in Google Cloud Spanner and how a better key layout can spread traffic evenly.

Why this matters

Spanner scales well, but it still depends on your table design. If many writes land on the same key range, you can create a hotspot: one split or one node receives disproportionate traffic and becomes the bottleneck.

Key Takeaway
A hot split is usually caused by a poor leading key choice, a single low-cardinality key value, or monotonic write patterns.

What you will do

  • Understand the hotspot concept in Spanner.
  • Spot bad key patterns that create skew.
  • Design better keys that distribute writes.
  • Deploy and fix a schema to compare outcomes.

Step 1: Understanding Hotspots

In Spanner, rows are split by the primary key range. If many writes target the same key prefix, the system concentrates work into a single split.

Hotspot
A hotspot is a region of the keyspace receiving much more traffic than the rest. This often leads to throttling and lower throughput.
Why it happens
Monotonic keys, a single tenant, or a low-cardinality prefix can force many writes into one split.
Example
If every event is written with a key that starts with the same user_id and timestamp, the write load is concentrated on one key range.

Step 2: Identify Bad Key Patterns

Some schema patterns look convenient but can hurt Spanner performance at scale.

Bad Pattern 1
Using a monotonic primary key such as PRIMARY KEY(order_id) where order_id increases steadily.
Bad Pattern 2
Using a low-cardinality key like region or a single tenant ID as the leading key for high-volume writes.
SQL · Bad Schema
-- Bad: all writes pile into a single shard
CREATE TABLE UserEvents (
  user_id              STRING(36) NOT NULL,
  event_time           TIMESTAMP NOT NULL,
  event_type           STRING(32),
  payload              STRING(MAX),
) PRIMARY KEY (user_id, event_time);
  • Identify the bad patterns
    Spot why the schema is likely to become hot.
  • Note the write skew risk
    Explain how one key prefix can dominate the workload.

Step 3: Good Key Design

A better design spreads writes by introducing a high-cardinality or hashed component into the leading key.

Good Strategy
Use a partition key that has high cardinality, such as a tenant ID plus a hash or shard suffix.
Why it helps
The write load is distributed across many key ranges instead of being concentrated into one.
SQL · Better Schema
-- Better: spread writes across many shards
CREATE TABLE UserEvents (
  tenant_id            STRING(16) NOT NULL,
  shard_id             INT64 NOT NULL,
  event_time           TIMESTAMP NOT NULL,
  event_type           STRING(32),
  payload              STRING(MAX),
) PRIMARY KEY (tenant_id, shard_id, event_time);
  • Understand the good key design
    Learn how a hash or shard component prevents hotspots.

    Notice how the better schema improves write distribution.

Step 4: Deploy the Bad Schema

This version creates a hotspot because writes for a single active customer will go to one key range.

SQL · Deploying the Hotspot Schema

CREATE TABLE UserEvents (
  user_id STRING(36) NOT NULL,
  event_time TIMESTAMP NOT NULL,
  event_type STRING(32),
  payload STRING(MAX),
) PRIMARY KEY (user_id, event_time);

Step 5: Fix with a Better Schema

The remedy is to introduce a shard or hash component so that rows are spread throughout the keyspace.

SQL · Fixed Schema
-- Replace the table with a sharded design
CREATE TABLE UserEvents (
  tenant_id STRING(16) NOT NULL,
  shard_id INT64 NOT NULL,
  event_time TIMESTAMP NOT NULL,
  event_type STRING(32),
  payload STRING(MAX),
) PRIMARY KEY (tenant_id, shard_id, event_time);
Why the fix works
The additional key component introduces many possible prefixes and enables more even distribution across splits.

    Replace the hot table design with a sharded key.
  • Compare the trade-off
    Notice that the schema is still easy to query while better distributed.
